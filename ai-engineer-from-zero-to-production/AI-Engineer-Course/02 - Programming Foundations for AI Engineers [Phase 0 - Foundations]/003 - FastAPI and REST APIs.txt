# FastAPI and REST APIs

## Overview
FastAPI has become the dominant web framework for AI microservices because it combines Python type hints with automatic OpenAPI documentation, async support, and a dependency injection system that maps naturally onto AI pipeline components. This lesson teaches you how to structure a FastAPI application properly — routing, request/response models, middleware, and the request lifecycle.

## Key Points
- FastAPI uses Python type annotations to automatically validate incoming requests and generate OpenAPI (Swagger) docs at /docs — zero configuration required
- Dependency injection via `Depends()` is how you share resources like LLM clients, database connections, and config objects across route handlers without global state
- Response models (the `response_model` parameter on route decorators) filter and validate outgoing data — they prevent accidentally leaking internal fields to clients
- Use `APIRouter` to split routes across multiple files; this keeps large AI services maintainable as they grow past the initial prototype
- HTTP status codes matter: 200 for success, 201 for resource creation, 422 for validation errors (FastAPI raises these automatically), 500 for unexpected server errors — never return 200 with an error body
- Middleware is the correct place for cross-cutting concerns: request ID injection, latency logging, and authentication token verification

## Code Example
```python
from __future__ import annotations

import uuid
from typing import Annotated

import uvicorn
from fastapi import APIRouter, Depends, FastAPI, HTTPException, Request, status
from fastapi.middleware.cors import CORSMiddleware
from pydantic import BaseModel, Field

# -------------------------------------------------------------------
# Pydantic models — request and response shapes
# -------------------------------------------------------------------
class ChatRequest(BaseModel):
    message: str = Field(..., min_length=1, max_length=4096, description="User message")
    session_id: str | None = Field(default=None, description="Conversation session ID")


class ChatResponse(BaseModel):
    reply: str
    session_id: str
    model: str


class HealthResponse(BaseModel):
    status: str
    version: str


# -------------------------------------------------------------------
# Simulated LLM client — replace with real SDK in production
# -------------------------------------------------------------------
class LLMClient:
    def __init__(self, model: str = "gpt-4o-mini") -> None:
        self.model = model

    def complete(self, message: str) -> str:
        # In production: return openai_client.chat.completions.create(...)
        return f"Echo from {self.model}: {message}"


# -------------------------------------------------------------------
# Dependency — one shared client per application lifetime
# -------------------------------------------------------------------
_llm_client = LLMClient()

def get_llm_client() -> LLMClient:
    return _llm_client


# -------------------------------------------------------------------
# Router — separating chat routes from health routes
# -------------------------------------------------------------------
chat_router = APIRouter(prefix="/chat", tags=["chat"])
health_router = APIRouter(tags=["health"])


@health_router.get("/health", response_model=HealthResponse)
async def health_check() -> HealthResponse:
    return HealthResponse(status="ok", version="1.0.0")


@chat_router.post("/", response_model=ChatResponse, status_code=status.HTTP_200_OK)
async def chat(
    body: ChatRequest,
    client: Annotated[LLMClient, Depends(get_llm_client)],
) -> ChatResponse:
    if not body.message.strip():
        raise HTTPException(
            status_code=status.HTTP_422_UNPROCESSABLE_ENTITY,
            detail="Message cannot be blank.",
        )
    reply = client.complete(body.message)
    session_id = body.session_id or str(uuid.uuid4())
    return ChatResponse(reply=reply, session_id=session_id, model=client.model)


# -------------------------------------------------------------------
# Application factory — prefer factory functions over module-level app
# -------------------------------------------------------------------
def create_app() -> FastAPI:
    app = FastAPI(
        title="AI Chat Service",
        description="Minimal FastAPI service wrapping an LLM",
        version="1.0.0",
    )

    # CORS — required when a browser front-end calls this API
    app.add_middleware(
        CORSMiddleware,
        allow_origins=["*"],   # lock this down in production
        allow_methods=["*"],
        allow_headers=["*"],
    )

    # Request-ID middleware for distributed tracing
    @app.middleware("http")
    async def attach_request_id(request: Request, call_next):
        request_id = str(uuid.uuid4())
        response = await call_next(request)
        response.headers["X-Request-ID"] = request_id
        return response

    app.include_router(health_router)
    app.include_router(chat_router)
    return app


app = create_app()

if __name__ == "__main__":
    uvicorn.run("003 - FastAPI and REST APIs:app", host="0.0.0.0", port=8000, reload=True)
```

## Notes
A common mistake is putting the LLM client instantiation inside the route handler — this creates a new client object (and potentially a new HTTP connection pool) on every request, which is extremely wasteful. Always instantiate expensive resources once at startup using the dependency injection system or a lifespan context manager (`@asynccontextmanager` passed to `FastAPI(lifespan=...)`). The `response_model` parameter is also frequently skipped, but it is your last line of defense against accidentally serializing internal fields (like API keys stored on a model object) into the HTTP response.

## Summary
You can now structure a FastAPI application with proper routing, dependency injection, and typed request/response contracts — the exact pattern used in production AI microservices at scale.
