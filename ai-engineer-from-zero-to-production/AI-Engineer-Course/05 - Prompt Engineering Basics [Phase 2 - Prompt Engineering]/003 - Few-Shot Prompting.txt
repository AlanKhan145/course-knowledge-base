# Few-Shot Prompting

## Overview
Few-shot prompting extends zero-shot by including a small number of worked input-output examples directly inside the prompt. These examples act as implicit demonstrations that teach the model the exact format, tone, reasoning style, and edge-case handling you require — without any fine-tuning. It is one of the highest-leverage techniques available to AI Engineers for improving output quality on specialized tasks.

## Key Points
- Few-shot examples are sampled from your actual data, not invented: synthetic or unrealistic examples teach the wrong distribution and can make performance worse.
- Example quality beats example quantity: 3 well-chosen, diverse examples typically outperform 10 redundant ones, and more examples increase token cost and latency.
- Examples should cover the full range of input variation — edge cases, ambiguous inputs, and the most common case — not just easy "textbook" inputs.
- The order of examples matters: models exhibit recency bias, so place your most representative or important example last, just before the live input.
- Label consistency is critical: if your output labels are "POSITIVE"/"NEGATIVE", never mix in "positive"/"negative" or "Positive"/"Negative" — inconsistency in examples degrades classification accuracy measurably.
- Few-shot tokens count against the context window and add cost to every request; for high-volume APIs, consider whether fine-tuning is more economical once you have 50+ stable examples.

## Code Example
```python
# few_shot_prompting.py
# Demonstrates few-shot prompting with structured examples.
# Requirements: pip install openai python-dotenv

import os
from openai import OpenAI
from dotenv import load_dotenv

load_dotenv()
client = OpenAI(api_key=os.getenv("OPENAI_API_KEY"))


# --- Define reusable few-shot examples as data, not hardcoded strings ---
# Storing examples as structured dicts makes them easy to manage, swap, and version.
SENTIMENT_EXAMPLES = [
    {
        "input": "I waited 45 minutes and nobody helped me. Absolute disaster.",
        "output": "NEGATIVE"
    },
    {
        "input": "The product arrived on time and works perfectly.",
        "output": "POSITIVE"
    },
    {
        "input": "It's fine. Does what it says on the box.",
        "output": "NEUTRAL"
    },
]


def build_few_shot_messages(
    task_instruction: str,
    examples: list[dict],
    live_input: str,
) -> list[dict]:
    """
    Build the messages list for a few-shot chat completion call.

    Each example is represented as a user/assistant turn pair, which is
    the canonical way to do few-shot with chat models (vs. text-davinci style).

    Args:
        task_instruction: The overall task description (goes in the system prompt).
        examples: List of dicts with 'input' and 'output' keys.
        live_input: The actual input to classify/process.

    Returns:
        A messages list ready to pass to client.chat.completions.create().
    """
    messages = [
        {"role": "system", "content": task_instruction}
    ]

    # Inject examples as alternating user/assistant turns
    for example in examples:
        messages.append({"role": "user", "content": example["input"]})
        messages.append({"role": "assistant", "content": example["output"]})

    # Add the live input as the final user turn
    messages.append({"role": "user", "content": live_input})

    return messages


def few_shot_classify(
    live_input: str,
    examples: list[dict] = SENTIMENT_EXAMPLES,
    model: str = "gpt-4o-mini",
) -> str:
    """Run a few-shot classification and return the raw label."""
    task_instruction = (
        "You are a sentiment classifier. "
        "Classify each customer review as exactly one of: POSITIVE, NEGATIVE, NEUTRAL. "
        "Respond with only the label."
    )
    messages = build_few_shot_messages(task_instruction, examples, live_input)

    response = client.chat.completions.create(
        model=model,
        messages=messages,
        temperature=0,
        max_tokens=10,   # Labels are short — cap tokens to reduce cost and hallucination
    )
    return response.choices[0].message.content.strip()


# --- Test cases ---
test_reviews = [
    "This is the worst purchase I have ever made.",           # Clear negative
    "Honestly not sure if I'd buy again, but it worked OK.", # Ambiguous / neutral
    "Incredible quality, fast shipping — highly recommend!", # Clear positive
    "The color was wrong but customer service fixed it fast." # Mixed — good test
]

print("Few-Shot Sentiment Classification Results")
print("=" * 45)
for review in test_reviews:
    label = few_shot_classify(review)
    print(f"Review : {review[:60]}...")
    print(f"Label  : {label}\n")


# --- Inspect the exact prompt being sent (debugging best practice) ---
def inspect_prompt(live_input: str) -> None:
    """Print the full message list so you can review what the model receives."""
    task_instruction = (
        "You are a sentiment classifier. "
        "Classify each customer review as exactly one of: POSITIVE, NEGATIVE, NEUTRAL. "
        "Respond with only the label."
    )
    messages = build_few_shot_messages(task_instruction, SENTIMENT_EXAMPLES, live_input)
    for i, msg in enumerate(messages):
        print(f"[{i}] role={msg['role']}\n    {msg['content']}\n")

inspect_prompt("This product changed my life.")
```

## Notes
When few-shot examples are stored as hardcoded strings, they become invisible technical debt — a change to one example breaks behavior in ways that are hard to trace. The data-driven approach shown above (examples as dicts) lets you version examples in a database or YAML file and A/B test different example sets. In production, few-shot prompts should be evaluated with automated metrics (F1, BLEU, exact match) on a held-out test set whenever examples are added or changed — treat them like test fixtures. Also be aware that very long few-shot prompts can push important context out of the model's effective attention window on some architectures; keep total prompt length under 50% of the model's context limit to leave room for long user inputs.

## Summary
Few-shot prompting is the fastest path to high-accuracy specialized behavior without fine-tuning — the key discipline is treating your examples as versioned, tested data assets rather than ad hoc strings embedded in code.
