# Context Window

## Overview
The context window is the total number of tokens an LLM can process in a single forward pass — encompassing your system prompt, conversation history, retrieved documents, tool outputs, and the model's own response. It is the single most important capacity constraint you will manage as an AI Engineer, affecting retrieval strategy, cost, latency, and system architecture.

## Key Points
- The context window is measured in tokens and is a hard limit. As of 2024-2025, common limits are: GPT-4o at 128k tokens, Claude 3.5 Sonnet at 200k tokens, Gemini 1.5 Pro at 1 million tokens. Exceeding this limit causes an API error — the call fails entirely.
- "Lost in the middle" is a well-documented phenomenon: LLMs recall information placed at the very beginning or very end of a long context significantly better than information placed in the middle. Placing your most critical instructions or retrieved chunks at the start or end of the prompt is therefore a practical engineering decision, not just style.
- Input tokens and output tokens both count toward cost but are billed separately, with output tokens typically costing 3-5x more per token than input tokens. A 128k context window does not mean free — filling it costs money on every call.
- Context window size affects latency. Time-to-first-token (TTFT) increases with prompt length. A 100k-token prompt takes measurably longer to process than a 1k-token prompt, which matters for user-facing applications.
- Statelessness is a critical architectural fact: LLMs have no memory between API calls. The context window IS the model's entire working memory for a given request. Conversation history must be explicitly managed and sent with every request.
- Practical context management strategies include: sliding window (drop oldest messages), summarization (compress old turns into a summary), RAG (retrieve only relevant chunks), and structured trimming (always preserve system prompt and latest user turn).

## Code Example
```python
# Context window management: sliding window with token-aware trimming
# Run: pip install tiktoken openai

import tiktoken
from openai import OpenAI

client = OpenAI()  # Reads OPENAI_API_KEY from environment

MODEL = "gpt-4o"
MAX_CONTEXT_TOKENS = 128_000
# Reserve tokens for the model's response
RESPONSE_BUDGET = 2_000
# Token budget available for messages
MESSAGE_BUDGET = MAX_CONTEXT_TOKENS - RESPONSE_BUDGET

enc = tiktoken.encoding_for_model(MODEL)

def count_message_tokens(messages: list[dict]) -> int:
    """Count tokens across all messages including role overhead."""
    total = 3  # reply primer
    for msg in messages:
        total += 3  # per-message overhead
        total += len(enc.encode(msg["role"]))
        total += len(enc.encode(msg["content"]))
    return total

def trim_messages_to_budget(
    system_msg: dict,
    history: list[dict],
    budget: int
) -> list[dict]:
    """
    Trim conversation history to fit within token budget.
    Always preserves: system prompt + most recent user message.
    Drops oldest messages first (sliding window strategy).
    """
    # System prompt is non-negotiable
    system_tokens = count_message_tokens([system_msg])
    remaining_budget = budget - system_tokens

    if not history:
        return [system_msg]

    # Always keep the last user message
    last_user_msg = history[-1]
    last_msg_tokens = count_message_tokens([last_user_msg])

    if last_msg_tokens > remaining_budget:
        raise ValueError(
            f"Last user message alone ({last_msg_tokens} tokens) "
            f"exceeds budget ({remaining_budget} tokens). "
            "Consider summarizing or chunking the input."
        )

    # Fill remaining budget with history, newest first
    kept_history = [last_user_msg]
    remaining_budget -= last_msg_tokens

    for msg in reversed(history[:-1]):
        msg_tokens = count_message_tokens([msg])
        if msg_tokens <= remaining_budget:
            kept_history.insert(0, msg)
            remaining_budget -= msg_tokens
        else:
            # No more room — drop this and all older messages
            print(f"[Context trim] Dropped {len(history) - len(kept_history)} old messages.")
            break

    return [system_msg] + kept_history

# --- Demo usage ---
system = {"role": "system", "content": "You are a concise technical assistant."}

history = [
    {"role": "user",      "content": "What is a transformer?"},
    {"role": "assistant", "content": "A transformer is a neural network architecture using self-attention."},
    {"role": "user",      "content": "What is RLHF?"},
    {"role": "assistant", "content": "RLHF is Reinforcement Learning from Human Feedback, used to align LLMs."},
    {"role": "user",      "content": "How large is GPT-4?"},
]

trimmed = trim_messages_to_budget(system, history, budget=MESSAGE_BUDGET)
total_tokens = count_message_tokens(trimmed)

print(f"Messages after trimming: {len(trimmed)}")
print(f"Total tokens to be sent: {total_tokens:,} / {MESSAGE_BUDGET:,} budget")
print(f"Context utilization:     {total_tokens / MAX_CONTEXT_TOKENS * 100:.2f}%")
```

## Notes
The 1-million-token context window of Gemini 1.5 Pro is architecturally impressive but does not make retrieval-augmented generation (RAG) obsolete. Filling a 1M-token context with documents costs roughly $2-4 per call at current pricing (as of 2025), which is prohibitive for high-frequency applications. More importantly, the "lost in the middle" degradation becomes more severe at extreme lengths. RAG remains the cost-efficient architecture for most production workloads; ultra-long context windows are most valuable for one-shot analysis tasks like processing an entire codebase or legal document. Always benchmark both approaches for your specific use case before committing to an architecture.

## Summary
The context window is the model's entire working memory, and managing it — through trimming, summarization, or retrieval — is a core responsibility of the AI Engineer. Understanding its hard limits, cost implications, and recall limitations lets you design systems that are both capable and economically viable.
