# JSON Schema

## Overview
JSON Schema is the industry-standard specification for describing the structure, types, and constraints of JSON data. In AI Engineering it acts as the contract between your application and the LLM — you tell the model exactly what shape its output must have, and your code can validate that contract at runtime. Understanding JSON Schema deeply is a prerequisite for Structured Output, Function Calling, and Tool Calling.

## Key Points
- JSON Schema is a JSON document itself; the `type`, `properties`, `required`, and `description` keywords are the ones you will use most in AI work
- The `description` field on each property is not decoration — LLMs read these descriptions to understand what value to place in each field; write them as if explaining to a junior developer
- Use `enum` to constrain a field to a fixed set of values, dramatically reducing model hallucination on categorical fields
- Nested objects and arrays are supported; keep nesting shallow (max 3 levels) to reduce model confusion and token cost
- `additionalProperties: false` tells strict-mode parsers to reject any keys the model invents beyond your schema, which is critical for security and stability
- Always mark truly required fields in the `required` array; optional fields should have sensible defaults documented in their description
- JSON Schema draft 2020-12 is the latest spec, but OpenAI's Structured Output uses a subset — avoid `$ref` cycles and some advanced keywords in that context

## Code Example
```python
import json
import jsonschema  # pip install jsonschema

# Define a schema for a structured course lesson object
lesson_schema = {
    "type": "object",
    "description": "A single lesson in an online course",
    "properties": {
        "title": {
            "type": "string",
            "description": "The lesson title, 5-10 words, sentence case"
        },
        "duration_minutes": {
            "type": "integer",
            "description": "Estimated duration in minutes (5-60)",
            "minimum": 5,
            "maximum": 60
        },
        "difficulty": {
            "type": "string",
            "enum": ["beginner", "intermediate", "advanced"],
            "description": "Difficulty level of the lesson"
        },
        "topics": {
            "type": "array",
            "description": "List of specific topics covered in this lesson",
            "items": {"type": "string"},
            "minItems": 1,
            "maxItems": 8
        },
        "has_code_example": {
            "type": "boolean",
            "description": "Whether the lesson includes a runnable code example"
        }
    },
    "required": ["title", "duration_minutes", "difficulty", "topics"],
    "additionalProperties": False
}

# A valid example payload
valid_lesson = {
    "title": "Introduction to JSON Schema",
    "duration_minutes": 20,
    "difficulty": "intermediate",
    "topics": ["JSON Schema syntax", "type constraints", "enum values", "required fields"],
    "has_code_example": True
}

# An invalid payload — duration is out of range, difficulty is not in enum
invalid_lesson = {
    "title": "Bad Lesson",
    "duration_minutes": 120,       # exceeds maximum of 60
    "difficulty": "expert",        # not in enum
    "topics": ["something"]
}

def validate_lesson(data: dict) -> bool:
    """Validate a lesson dict against the schema, printing errors."""
    validator = jsonschema.Draft202012Validator(lesson_schema)
    errors = list(validator.iter_errors(data))
    if errors:
        for err in errors:
            print(f"Validation error at '{'.'.join(str(p) for p in err.path)}': {err.message}")
        return False
    print("Valid!")
    return True

print("--- Valid lesson ---")
validate_lesson(valid_lesson)

print("\n--- Invalid lesson ---")
validate_lesson(invalid_lesson)

# Generate the schema string to embed in a prompt
print("\n--- Schema as prompt string ---")
print(json.dumps(lesson_schema, indent=2))
```

## Notes
The `description` fields carry more weight than most beginners expect — in GPT-4o and Claude 3.x experiments, adding precise, instruction-like descriptions to properties reduces field-level errors by 30-50% compared to using bare field names. Avoid deeply nested schemas when using OpenAI's Structured Output mode: the feature requires the entire schema to be inlined (no external `$ref` resolution), and very large schemas inflate your prompt token count significantly. Also note that `additionalProperties: false` is required by OpenAI's strict Structured Output mode — it is not optional if you want the model to be constrained to your exact field set.

## Summary
You can now write JSON Schemas that precisely describe the data contracts your AI pipelines require, including types, enums, constraints, and descriptions that guide both the model and your validation layer.
