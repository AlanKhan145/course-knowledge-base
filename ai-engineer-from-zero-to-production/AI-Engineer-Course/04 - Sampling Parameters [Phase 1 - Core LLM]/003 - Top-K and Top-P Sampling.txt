# Top-K and Top-P Sampling

## Overview
Top-K and Top-P (nucleus) sampling are vocabulary filtering techniques applied after temperature scaling but before the final token is sampled. They constrain the candidate pool to prevent the model from selecting very low-probability, incoherent tokens. Understanding both strategies — and their tradeoffs — allows AI Engineers to control output quality with more precision than temperature alone.

## Key Points
- Top-K sampling retains only the K highest-probability tokens at each step and redistributes probability mass across just those K candidates; all other tokens are zeroed out before sampling
- Top-P (nucleus) sampling retains the smallest set of tokens whose cumulative probability meets or exceeds a threshold P (e.g., 0.9); the candidate pool size varies dynamically based on the distribution shape
- Top-P is generally preferred over Top-K in practice because it adapts to distribution width: when the model is confident, the nucleus is small (few tokens); when uncertain, it expands automatically
- Using top-p=1.0 disables nucleus filtering entirely — the full vocabulary remains available; using top-p=0.1 creates a very narrow, near-greedy candidate set
- Top-K and Top-P are typically applied together with temperature; the standard production stack is: apply temperature, apply top-k, apply top-p, then sample
- A common high-quality starting configuration: temperature=0.7, top-p=0.9, top-k=50 — this balances diversity with coherence for most tasks

## Code Example
```python
import os
from anthropic import Anthropic

# Note: Anthropic's API exposes top_p and top_k directly.
# The example below shows their effect by comparing outputs.

client = Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])

PROMPT = (
    "List three creative names for a new coffee shop. "
    "Give only the names, one per line, no explanations."
)


def sample_with_params(top_p: float, top_k: int, temperature: float = 0.9) -> str:
    """Return raw text completion for given sampling parameters."""
    response = client.messages.create(
        model="claude-sonnet-4-5",
        max_tokens=80,
        temperature=temperature,
        top_p=top_p,
        top_k=top_k,
        messages=[{"role": "user", "content": PROMPT}],
    )
    return response.content[0].text.strip()


def demonstrate_top_p():
    """Show how top_p width affects output diversity."""
    print("=== Top-P Comparison (temperature=0.9, top_k=50) ===\n")
    for top_p in [0.1, 0.5, 0.95]:
        result = sample_with_params(top_p=top_p, top_k=50)
        print(f"top_p={top_p}:")
        print(result)
        print()


def demonstrate_top_k():
    """Show how top_k pool size affects output diversity."""
    print("=== Top-K Comparison (temperature=0.9, top_p=1.0) ===\n")
    for top_k in [1, 10, 200]:
        result = sample_with_params(top_p=1.0, top_k=top_k)
        print(f"top_k={top_k}:")
        print(result)
        print()


def show_combined_best_practice():
    """Demonstrate a balanced production-ready configuration."""
    print("=== Balanced Production Config (temp=0.7, top_p=0.9, top_k=50) ===\n")
    result = sample_with_params(top_p=0.9, top_k=50, temperature=0.7)
    print(result)


if __name__ == "__main__":
    demonstrate_top_p()
    demonstrate_top_k()
    show_combined_best_practice()
```

## Notes
A subtle but important point: Top-K with K=1 is identical to greedy decoding regardless of temperature, because only the single best token survives the filter. When both Top-K and Top-P are set, Top-K is typically applied first to create a shortlist, and then Top-P further narrows that shortlist — so they compound. In practice, many hosted APIs (including OpenAI and Anthropic) recommend not setting both to extreme values simultaneously, as this can create an empty candidate set and trigger undefined behavior. Also note that Top-P and Top-K do not exist in isolation on all inference backends: vLLM, llama.cpp, and Hugging Face Transformers implement them but with slightly different ordering conventions — always check the backend documentation.

## Summary
Top-K and Top-P are complementary filters that constrain the token candidate pool and prevent incoherent low-probability tokens from being selected. After this lesson you can configure both parameters deliberately for any task, understanding why Top-P is preferred for most production use cases.
