# Combining Sampling Parameters in Practice

## Overview
Individual sampling parameters are powerful, but real AI Engineering requires knowing how to combine them coherently for specific tasks. This lesson provides concrete parameter recipes for the most common production use cases — factual Q&A, creative generation, code completion, structured extraction, and conversational chat — and explains the reasoning behind each configuration choice.

## Key Points
- Parameters interact multiplicatively: temperature, top-k, and top-p all affect the same underlying token distribution, so cranking all three to extremes simultaneously produces unpredictable results
- The "greedy recipe" (temperature=0, top-p=1, top-k=1) is optimal for tasks where there is a single correct answer and variance is harmful — SQL generation, math, yes/no classification
- The "creative recipe" (temperature=0.9–1.1, top-p=0.95, top-k=50) is optimal for brainstorming, marketing copy, story writing, and diversity-of-ideas tasks
- The "balanced recipe" (temperature=0.3–0.5, top-p=0.9, top-k=40) is the best starting point for most conversational assistants and general-purpose chatbots
- Stop sequences and max_tokens should always be set regardless of the creativity level — they are operational safety nets, not stylistic choices
- Always run an A/B evaluation with at least 20–50 representative prompts before locking in a parameter configuration for production

## Code Example
```python
import os
from anthropic import Anthropic
from dataclasses import dataclass

client = Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])


@dataclass
class SamplingConfig:
    """Encapsulates a named sampling configuration for reuse."""
    name: str
    temperature: float
    top_p: float
    top_k: int
    max_tokens: int
    stop_sequences: list[str]


# ── Predefined recipes ────────────────────────────────────────────────────────

CONFIGS = {
    "factual": SamplingConfig(
        name="Factual / Deterministic",
        temperature=0.0,
        top_p=1.0,
        top_k=1,
        max_tokens=256,
        stop_sequences=[],
    ),
    "code": SamplingConfig(
        name="Code Generation",
        temperature=0.2,
        top_p=0.95,
        top_k=40,
        max_tokens=512,
        stop_sequences=["```\n\n", "# END"],
    ),
    "balanced": SamplingConfig(
        name="Conversational / Balanced",
        temperature=0.5,
        top_p=0.9,
        top_k=40,
        max_tokens=400,
        stop_sequences=[],
    ),
    "creative": SamplingConfig(
        name="Creative Writing",
        temperature=1.0,
        top_p=0.95,
        top_k=50,
        max_tokens=600,
        stop_sequences=[],
    ),
    "extraction": SamplingConfig(
        name="Structured Extraction",
        temperature=0.0,
        top_p=1.0,
        top_k=1,
        max_tokens=128,
        stop_sequences=["\n\n", "```"],
    ),
}


def run_with_config(prompt: str, config: SamplingConfig) -> dict:
    """Run a prompt with a named sampling config and return structured results."""
    response = client.messages.create(
        model="claude-sonnet-4-5",
        temperature=config.temperature,
        top_p=config.top_p,
        top_k=config.top_k,
        max_tokens=config.max_tokens,
        stop_sequences=config.stop_sequences if config.stop_sequences else None,
        messages=[{"role": "user", "content": prompt}],
    )
    return {
        "config": config.name,
        "stop_reason": response.stop_reason,
        "tokens": response.usage.output_tokens,
        "text": response.content[0].text.strip(),
    }


def compare_configs_on_prompt(prompt: str, config_keys: list[str]):
    """Compare multiple configs side-by-side for the same prompt."""
    print(f"Prompt: {prompt!r}\n" + "=" * 60)
    for key in config_keys:
        config = CONFIGS[key]
        result = run_with_config(prompt, config)
        print(f"\n[{result['config']}]")
        print(f"  tokens={result['tokens']} | stop={result['stop_reason']}")
        print(f"  {result['text'][:200]}")
    print()


if __name__ == "__main__":
    # 1. Factual vs creative on the same open-ended prompt
    compare_configs_on_prompt(
        "What could the year 2075 look like?",
        ["factual", "creative"],
    )

    # 2. Code generation config
    result = run_with_config(
        "Write a Python function that reverses a string.",
        CONFIGS["code"],
    )
    print(f"[{result['config']}]\n{result['text']}\n")

    # 3. Structured extraction config
    result = run_with_config(
        "Extract the city name from: 'I live in Berlin and love it here.'\n"
        "Reply with only the city name.",
        CONFIGS["extraction"],
    )
    print(f"[{result['config']}] Extracted: '{result['text']}'")
```

## Notes
One common mistake is using creative-level temperature for structured extraction tasks: the model may hallucinate fields or output malformed JSON. Conversely, using near-zero temperature for a brainstorming task will return almost the same ideas every call, defeating the purpose. A practical workflow is to define a SamplingConfig dataclass (or Pydantic model) per task type in your codebase, store configurations in a settings file or environment variable, and avoid hardcoding raw numbers at the API call site. This makes parameter tuning a configuration concern rather than a code change, which is critical for teams where prompts and parameters are iterated on frequently.

## Summary
Combining sampling parameters effectively means matching the configuration to the task's tolerance for variance. After this lesson you can select and justify a complete parameter recipe for any production use case rather than relying on a single global default.
