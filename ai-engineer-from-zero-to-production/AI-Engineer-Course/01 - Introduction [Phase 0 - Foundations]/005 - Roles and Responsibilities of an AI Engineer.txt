# Roles and Responsibilities of an AI Engineer

## Overview
This lesson maps out the concrete day-to-day responsibilities of an AI Engineer in a product organization — from initial prototyping to production monitoring. Understanding these responsibilities helps you know exactly what skills to prioritize and what you will be accountable for when working on a team.

## Key Points
- Prototyping and Feasibility Assessment: AI Engineers rapidly evaluate whether a given problem is solvable with current LLM capabilities — this involves building quick proof-of-concept prompts, testing with real data samples, and producing an honest feasibility report before a team commits engineering resources.
- Prompt and System Design: writing, versioning, and iterating on system prompts and prompt templates is a core daily activity — prompts are code; they must be stored in version control, tested against regressions, and reviewed like any other production artifact.
- Retrieval and Data Pipeline Ownership: AI Engineers own the pipelines that ingest, chunk, embed, and index documents into vector databases — a poorly designed retrieval pipeline is the most common reason a RAG system fails in production.
- Evaluation and Quality Assurance: building automated evaluation suites (using LLM-as-judge, reference-based metrics, or human rating rubrics) is an explicit responsibility — without evals, you cannot ship safely or iterate confidently.
- Observability and Incident Response: AI Engineers instrument LLM calls with tracing tools (LangSmith, Langfuse, OpenTelemetry), set up latency and cost dashboards, define alert thresholds, and respond to incidents where model outputs degrade or costs spike unexpectedly.

## Code Example
```python
# A realistic snapshot of AI Engineer daily work:
# A versioned prompt manager with basic eval harness

import json
import hashlib
from datetime import datetime
from openai import OpenAI

client = OpenAI()

# --- RESPONSIBILITY 1: Prompt versioning ---
PROMPT_REGISTRY = {
    "v1.0": {
        "system": "You are a customer support agent. Be concise and helpful.",
        "created": "2024-01-10",
    },
    "v1.1": {
        "system": (
            "You are a customer support agent for TechCorp. "
            "Be concise, empathetic, and always offer a next step. "
            "If unsure, say so and escalate."
        ),
        "created": "2024-02-15",
    },
}


def get_prompt(version: str = "v1.1") -> str:
    """Retrieve a versioned system prompt."""
    return PROMPT_REGISTRY[version]["system"]


# --- RESPONSIBILITY 2: Structured LLM call with tracing ---
def call_support_agent(user_message: str, prompt_version: str = "v1.1") -> dict:
    """
    Production-style LLM call that captures:
    - Which prompt version was used (for regression tracking)
    - Token usage (for cost monitoring)
    - Response latency
    """
    system_prompt = get_prompt(prompt_version)
    start = datetime.now()

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_message},
        ],
        temperature=0.3,
        max_tokens=256,
    )

    latency_ms = (datetime.now() - start).total_seconds() * 1000
    content = response.choices[0].message.content

    # Trace record — in production this goes to Langfuse or your observability backend
    trace = {
        "prompt_version": prompt_version,
        "input_hash": hashlib.md5(user_message.encode()).hexdigest(),
        "tokens_in": response.usage.prompt_tokens,
        "tokens_out": response.usage.completion_tokens,
        "latency_ms": round(latency_ms, 1),
        "output": content,
    }
    return trace


# --- RESPONSIBILITY 3: Automated evaluation ---
EVAL_DATASET = [
    {
        "input": "My order hasn't arrived after 2 weeks.",
        "must_contain": ["sorry", "escalate", "next step"],
    },
    {
        "input": "How do I reset my password?",
        "must_contain": ["password", "reset"],
    },
]


def run_eval(prompt_version: str) -> dict:
    """Simple keyword-based eval — replace with LLM-as-judge in production."""
    results = []
    for case in EVAL_DATASET:
        trace = call_support_agent(case["input"], prompt_version)
        output_lower = trace["output"].lower()
        passed = all(kw in output_lower for kw in case["must_contain"])
        results.append({"input": case["input"], "passed": passed, "output": trace["output"]})

    pass_rate = sum(1 for r in results if r["passed"]) / len(results)
    return {"prompt_version": prompt_version, "pass_rate": pass_rate, "details": results}


if __name__ == "__main__":
    # Compare two prompt versions
    for version in ["v1.0", "v1.1"]:
        report = run_eval(version)
        print(f"Version {version}: pass_rate={report['pass_rate']:.0%}")
```

## Notes
In most product companies, AI Engineers do not work in isolation — they collaborate closely with product managers (to define what "good" output means), backend engineers (for API integration and data pipelines), and sometimes legal/compliance teams (for content moderation and privacy). The ownership model varies: at some companies AI Engineers own the full stack from data ingestion to the deployed endpoint; at others they hand off to platform teams for infrastructure. The single most underestimated responsibility in this list is evaluation — teams that skip structured evals end up doing informal vibe-checks and shipping regressions silently.

## Summary
An AI Engineer's responsibilities span the entire lifecycle of an AI feature — from feasibility assessment through prompt design, retrieval pipeline ownership, evaluation, and production observability — making it a broad, high-accountability engineering role.
