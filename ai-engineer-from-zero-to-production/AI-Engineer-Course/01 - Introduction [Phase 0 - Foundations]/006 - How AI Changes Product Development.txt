# How AI Changes Product Development

## Overview
The integration of LLMs into product development is not just an additive feature — it changes how products are designed, how quickly they can be iterated, and what kinds of user experiences are now feasible. This lesson examines the structural shifts AI introduces to the product development lifecycle and how AI Engineers operate within them.

## Key Points
- AI compresses prototyping cycles dramatically: a feature that previously required months of custom ML work (training data collection, labeling, model training, serving) can now be prototyped in hours using an LLM API, shifting the bottleneck from implementation to evaluation and product definition.
- Product specs must now include "AI behavior specs" — explicit descriptions of desired model behavior, edge case handling, and output format, because non-deterministic outputs mean the same input can produce different results across calls; this requires a new class of documentation and testing.
- User experience design changes fundamentally: AI-powered interfaces (chat, copilots, summarizers) require designing for graceful degradation when the model is wrong, which means AI Engineers and UX designers must collaborate on error states, confidence signaling, and user correction flows.
- Cost is a first-class product constraint: unlike traditional software where marginal cost per request approaches zero, each LLM API call has a direct monetary cost (tokens in + tokens out); AI Engineers must architect for cost efficiency from the start, not as an afterthought.
- AI enables new product categories that were previously impossible at startup scale: personalized tutors, document intelligence, code generation tools, voice assistants — but each requires AI-specific quality assurance that traditional QA teams are not trained to perform.

## Code Example
```python
# Demonstrating how AI changes product iteration speed:
# A traditional rule-based classifier vs. an AI-powered one

import time
import os
from openai import OpenAI

client = OpenAI()


# --- TRADITIONAL APPROACH (weeks to build, brittle) ---
def rule_based_support_router(ticket: str) -> str:
    """
    Old approach: keyword rules written by hand.
    Takes days/weeks to cover edge cases, breaks with typos or paraphrasing.
    """
    ticket_lower = ticket.lower()
    if any(word in ticket_lower for word in ["refund", "money back", "charge"]):
        return "billing"
    elif any(word in ticket_lower for word in ["password", "login", "access"]):
        return "auth"
    elif any(word in ticket_lower for word in ["crash", "bug", "error", "broken"]):
        return "technical"
    else:
        return "general"


# --- AI-POWERED APPROACH (hours to build, robust) ---
def ai_support_router(ticket: str) -> dict:
    """
    New approach: LLM-based classification.
    Handles paraphrasing, typos, multilingual input out of the box.
    Trade-off: ~$0.0001 per call, ~300ms latency.
    """
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {
                "role": "system",
                "content": (
                    "Classify the support ticket into exactly one category: "
                    "billing, auth, technical, general. "
                    'Respond ONLY with valid JSON: {"category": "...", "confidence": 0.0-1.0}'
                ),
            },
            {"role": "user", "content": f"Ticket: {ticket}"},
        ],
        temperature=0,
        response_format={"type": "json_object"},
    )
    import json
    return json.loads(response.choices[0].message.content)


# --- COST AWARENESS (critical for product teams) ---
def estimate_monthly_cost(
    avg_input_tokens: int,
    avg_output_tokens: int,
    requests_per_day: int,
    price_per_1k_input: float = 0.00015,   # gpt-4o-mini pricing (2024)
    price_per_1k_output: float = 0.0006,
) -> dict:
    """Calculate monthly LLM cost — every AI Engineer should do this before launch."""
    daily_cost = (
        (avg_input_tokens / 1000 * price_per_1k_input) +
        (avg_output_tokens / 1000 * price_per_1k_output)
    ) * requests_per_day
    return {
        "daily_cost_usd": round(daily_cost, 4),
        "monthly_cost_usd": round(daily_cost * 30, 2),
        "annual_cost_usd": round(daily_cost * 365, 2),
    }


if __name__ == "__main__":
    ticket = "I was charged twice last month and cant get into my account to check"

    # Traditional router
    t_start = time.time()
    rule_result = rule_based_support_router(ticket)
    print(f"Rule-based: {rule_result} ({(time.time()-t_start)*1000:.0f}ms)")

    # AI router
    t_start = time.time()
    ai_result = ai_support_router(ticket)
    print(f"AI-based: {ai_result} ({(time.time()-t_start)*1000:.0f}ms)")

    # Cost projection for a mid-size product
    cost = estimate_monthly_cost(
        avg_input_tokens=150,
        avg_output_tokens=30,
        requests_per_day=10_000,
    )
    print(f"\nCost projection (10k req/day): {cost}")
```

## Notes
One of the biggest cultural changes AI introduces is the shift from deterministic testing to probabilistic evaluation. Traditional QA tests pass or fail definitively; AI feature tests pass with a score. This requires product teams to agree on acceptance thresholds ("we ship when eval pass rate exceeds 90%") before development starts, not after. Another real-world gotcha: AI features often degrade silently after launch — the model provider may update the underlying model, or the distribution of user inputs shifts over time — so continuous monitoring is non-optional. Finally, legal and compliance teams are increasingly involved in AI feature launches; build approval workflows into your release process from day one.

## Summary
AI fundamentally changes product development by compressing build cycles while introducing new requirements around probabilistic quality assurance, cost management, and continuous monitoring — AI Engineers sit at the intersection of all three.
