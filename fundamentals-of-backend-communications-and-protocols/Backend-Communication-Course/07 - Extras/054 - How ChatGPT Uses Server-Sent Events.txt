# 054 - How ChatGPT Uses Server-Sent Events

Section: 07 - Extras
Duration: 44min

---

## 1. Summary

This lecture analyzes how ChatGPT streams responses to the browser using a technique similar to **Server-Sent Events**, or SSE.

The instructor opens ChatGPT in the browser, inspects the Network tab, and studies how the frontend communicates with the backend. The main focus is the request that sends a user message to the conversation endpoint and receives a streaming response.

The lecture shows that ChatGPT sends a request with an `Accept: text/event-stream` header and receives a response with `Content-Type: text/event-stream`. This allows the backend to send partial chunks of the model's answer as they are generated.

Instead of waiting for the entire answer to finish, the browser receives small pieces of text over time and renders them progressively. This is what creates the familiar typing effect.

The lecture also discusses related backend details such as TLS, HTTP/2, Cloudflare, access tokens, conversation IDs, message IDs, parent message IDs, moderation calls, title generation, and feedback APIs.

The most important backend concept is this:

Server-Sent Events are still based on HTTP request-response, but the response is long-lived and streamed in chunks.

---

## 2. Key Concepts

### Server-Sent Events

Server-Sent Events, or SSE, allow a server to push updates to the browser over a single HTTP response.

The client sends a request.

The server keeps the response open.

Then the server sends events or chunks of text over time.

This is useful when the server needs to stream data gradually, such as:

* AI model tokens,
* logs,
* notifications,
* progress updates,
* live status updates.

---

### SSE Is Still Request-Response

SSE is not the same as WebSocket.

With SSE, the client still sends a normal HTTP request.

The difference is that the response does not finish immediately.

Instead, the server keeps writing data to the response.

So SSE can be thought of as:

```text id="39uyhv"
Client sends one request
Server sends a long streaming response
Client reads chunks as they arrive
```

This fits AI text generation very well because the model generates output progressively.

---

### ChatGPT Streaming Behavior

In the lecture, when the user sends a message to ChatGPT, the browser sends a `POST` request to a conversation endpoint.

The request includes a header similar to:

```text id="qpcgin"
Accept: text/event-stream
```

The response includes:

```text id="xq4443"
Content-Type: text/event-stream
```

This means the browser expects a streaming response.

As the model generates text, the backend sends partial chunks back to the frontend.

The frontend reads those chunks and updates the UI gradually.

---

### Why ChatGPT Appears to Type

ChatGPT does not wait for the whole answer to be ready.

Instead, it receives small pieces of the answer from the server.

The browser then appends these pieces to the visible message.

That is why the response appears to be typed live.

The actual flow is:

```text id="3hjbyh"
User sends prompt
Backend forwards prompt to model
Model generates tokens/chunks
Backend streams chunks to browser
Browser renders chunks progressively
```

---

### EventSource vs Custom Streaming

Browsers provide a built-in API called `EventSource` for SSE.

However, the lecture observes that ChatGPT may not use the browser's built-in `EventSource` API.

Instead, it appears to use a custom streaming approach, likely based on `fetch()` and a readable stream.

This means the browser manually reads streamed chunks from the response body.

A simplified version of this idea is:

```javascript id="2tthxh"
const response = await fetch("/api/conversation", {
  method: "POST",
  headers: {
    "Accept": "text/event-stream",
    "Content-Type": "application/json"
  },
  body: JSON.stringify({
    message: "Hello"
  })
});

const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
  const { value, done } = await reader.read();

  if (done) break;

  const chunk = decoder.decode(value, { stream: true });
  console.log(chunk);
}
```

This allows the frontend to process data as it arrives instead of waiting for the full response.

---

### HTTP/2 and Streaming

The lecture observes that the browser negotiated HTTP/2.

HTTP/2 is helpful for SSE-style traffic because it supports multiplexing.

With HTTP/1.1, browsers have a limited number of connections per domain. A long-lived SSE response can occupy one of those connections.

With HTTP/2, many streams can share the same TCP connection.

This allows the browser to load JavaScript, fonts, API calls, and streaming responses more efficiently over fewer connections.

---

### TLS 1.3

The lecture also observes that the browser uses TLS 1.3.

TLS protects the communication between the browser and the server.

The browser and server perform a TLS handshake, agree on encryption parameters, and then exchange encrypted HTTP traffic.

In the inspected request, the browser negotiates a secure connection before sending application data.

---

### Cloudflare as Reverse Proxy / CDN

The lecture mentions Cloudflare sitting in front of ChatGPT.

Cloudflare can act as:

* CDN,
* reverse proxy,
* TLS terminator,
* DDoS protection layer,
* bot protection layer,
* CAPTCHA provider.

Before the main application loads, Cloudflare may redirect, inspect, or challenge the client.

This is why the instructor sees redirects and some forbidden/CAPTCHA-related requests.

### Session Request

Before sending actual chat messages, the frontend makes a session request.

This request returns information about the current authenticated user/session.

It may include data such as:

* user information,
* access token,
* session state,
* authentication metadata.

The lecture suggests that cookies may contain refresh/session information, and the session endpoint returns a temporary access token used for authenticated API calls.

---

### Conversation List Endpoint

The frontend also requests the list of conversations.

The lecture observes a request with pagination fields like:

```text id="eaofhh"
offset=0
limit=20
```

This suggests the frontend loads conversations in pages.

The instructor discusses the trade-offs of offset-based pagination, especially if the dataset becomes large.

---

### Conversation ID

Each chat conversation has a unique conversation ID.

A conversation acts like a container for a chain of messages.

When the user opens or continues a chat, the frontend references this conversation ID.

This allows the backend to know which conversation the message belongs to.

---

### Message ID

Each message also has a unique message ID.

The client generates or sends a message ID with each user message.

This allows the system to track individual messages inside a conversation.

---

### Parent Message ID

Each message may include a `parent_message_id`.

This links the current message to the previous message.

This is important for context.

For example:

```text id="b0a0lg"
User: What is 1 + 1?
Assistant: 1 + 1 = 2.
User: No, that's wrong.
```

The last message only makes sense if the system knows which previous message it refers to.

The parent message ID helps build a message chain.

---

### Message Chain

A conversation is not just one isolated request.

It is a chain of linked messages.

The backend can use this chain to reconstruct the context of the conversation.

This is why ChatGPT can answer follow-up questions such as:

```text id="vlfx68"
Can you rewrite that using fetch?
```

The word "that" refers to the previous answer.

---

### Model Slug

The lecture observes that the frontend receives information about available models.

Then future conversation requests include a model identifier, sometimes called a model slug.

The model slug tells the backend which model should answer the request.

---

### Moderation Request

The lecture also observes moderation requests.

After a message is sent, the frontend/backend may trigger a moderation call.

The moderation request includes identifiers such as:

* conversation ID,
* message ID.

This suggests that messages may be checked for policy or safety reasons.

---

### Title Generation

The lecture observes an endpoint for generating a conversation title.

When a new chat starts, the chat does not immediately have a meaningful title.

After the user sends a message, the system can generate a title based on the conversation content.

For example, if the user says:

```text id="hsdocq"
Hi
```

the system may generate a simple title like:

```text id="v65261"
Greeting and Help Request
```

This title generation may itself use a model.

---

### Feedback API

The lecture also inspects what happens when the user likes or dislikes a response.

When the user clicks thumbs up or thumbs down, the frontend sends a feedback request.

The request includes:

* conversation ID,
* message ID,
* rating,
* optional text feedback,
* optional tags.

This allows the system to associate feedback with a specific assistant message.

---

### UUID vs ULID Discussion

The instructor notices that many identifiers appear random.

He discusses the possible database performance trade-offs of random UUIDs.

Random IDs can cause random reads and writes in some database storage engines.

He also mentions ULID, a lexicographically sortable identifier.

ULIDs can be useful when records are usually created and read in time order.

This part is mostly architectural speculation, but it is useful for thinking about backend design.

## 3. Important Notes

### SSE Works Well for AI Responses

AI model responses are generated gradually.

SSE is a natural fit because the server can stream partial output as soon as it is available.

This improves perceived latency.

The user sees the answer begin quickly instead of waiting for the full answer.

---

### SSE Is One-Way

SSE is mainly server-to-client streaming.

The client sends a request first, but after that, the server streams data back.

If the client wants to send another message, it sends another HTTP request.

This is different from WebSocket, where both sides can send messages anytime over the same connection.

---

### ChatGPT Does Not Need WebSocket for Basic Streaming

For normal ChatGPT responses, WebSocket is not required.

The browser sends a prompt.

The server streams the answer.

This is a perfect use case for SSE or fetch-based streaming.

WebSocket would be useful for fully bidirectional real-time communication, but simple AI answer streaming does not require it.

---

### HTTP/2 Helps with Long-Lived Streams

Because SSE responses can stay open for a long time, HTTP/2 is useful.

HTTP/2 allows multiple streams over one connection.

This avoids the connection-limit problems of HTTP/1.1.

---

### DevTools May Not Always Show the Stream Clearly

The lecture notes that Chrome DevTools may not display the stream in the EventStream tab if the site does not use the built-in `EventSource` browser API.

If the frontend uses `fetch()` with `ReadableStream`, the stream may appear differently.

This means DevTools behavior depends on the implementation.

---

### Streaming with Fetch Requires Manual Reading

Normal fetch usage often looks like this:

```javascript id="dozln1"
const data = await response.json();
```

or:

```javascript id="bd23x4"
const text = await response.text();
```

But these wait until the full response is complete.

For streaming, the frontend must use:

```javascript id="h2jbw3"
response.body.getReader()
```

This allows the frontend to read chunks as they arrive.

---

### Parent Message IDs Enable Context

The parent message ID is critical for conversation continuity.

Without it, follow-up questions would be harder to understand.

For example:

```text id="ag3szr"
Can you rewrite that?
```

This only makes sense if the backend knows what "that" refers to.

---

### Access Tokens Can Expire

The lecture shows the instructor encountering forbidden errors after being idle.

This suggests the access token may expire quickly.

When that happens, the client may need to refresh the session or request a new token.

This is common in authenticated web applications.

---

### Conversation Titles Can Be Generated Automatically

The lecture highlights that conversation titles may be generated after the first user message.

This is a clever UX detail.

Instead of asking the user to name every chat, the system can summarize the conversation and create a title automatically.

---

### Feedback Is Tied to Specific Messages

Feedback is not only tied to a conversation.

It is tied to a specific assistant response.

This is important because one conversation can contain many assistant messages, and each message may have different quality.

## 4. Examples

### Example 1: Basic SSE Flow

```text id="p4yzs5"
Client sends POST /conversation
Client includes Accept: text/event-stream
Server receives prompt
Model starts generating answer
Server streams chunks back
Client renders chunks progressively
Server closes stream when answer is done
```

---

### Example 2: Headers for Streaming

Request:

```http id="jn60cc"
POST /backend-api/conversation HTTP/2
Accept: text/event-stream
Content-Type: application/json
Authorization: Bearer <access_token>
```

Response:

```http id="quv0o6"
HTTP/2 200 OK
Content-Type: text/event-stream
```

---

### Example 3: Fetch-Based Streaming

```javascript id="tttau9"
async function streamResponse() {
  const response = await fetch("/api/conversation", {
    method: "POST",
    headers: {
      "Accept": "text/event-stream",
      "Content-Type": "application/json"
    },
    body: JSON.stringify({
      message: "Can you explain SSE?"
    })
  });

  const reader = response.body.getReader();
  const decoder = new TextDecoder();

  while (true) {
    const { value, done } = await reader.read();

    if (done) {
      break;
    }

    const chunk = decoder.decode(value, { stream: true });
    console.log("Received chunk:", chunk);
  }
}
```

This approach does not use the browser's `EventSource` API.

It manually reads streamed chunks from the response body.

---

### Example 4: Message Payload

A simplified conversation request may look like this:

```json id="uz17mc"
{
  "conversation_id": "conversation_123",
  "message": {
    "id": "message_456",
    "role": "user",
    "content": {
      "content_type": "text",
      "parts": [
        "Can you explain Server-Sent Events?"
      ]
    }
  },
  "parent_message_id": "message_455",
  "model": "model_slug"
}
```

Important fields:

* `conversation_id`: identifies the chat.
* `message.id`: identifies the current message.
* `parent_message_id`: links this message to the previous message.
* `model`: tells the backend which model to use.
* `content.parts`: contains the user text.

---

### Example 5: Follow-Up Question

User sends:

```text id="u53ul2"
Can you write JavaScript code for SSE without EventSource?
```

Assistant responds with code.

Then user sends:

```text id="v1klph"
Can you rewrite that using fetch?
```

The second message depends on the first answer.

The backend can understand the follow-up because the new message points to the previous message using `parent_message_id`.

---

### Example 6: Feedback Request

A simplified feedback request may look like this:

```json id="z3mmek"
{
  "conversation_id": "conversation_123",
  "message_id": "message_789",
  "rating": "thumbs_up",
  "text": "Great answer"
}
```

This tells the backend which exact assistant message received feedback.

---

### Example 7: Title Generation

After a new conversation starts, the frontend may call a title generation endpoint.

Simplified flow:

```text id="lt3xht"
User sends first message
Assistant responds
Frontend calls generate-title endpoint
Backend generates short title
Sidebar updates with new conversation title
```

Example:

```text id="7xumir"
User message: "Can you explain JavaScript SSE?"
Generated title: "JavaScript SSE Explanation"
```

---

### Example 8: HTTP/2 Multiplexing

With HTTP/2, multiple requests can share one connection:

```text id="wzllv4"
Connection 1:
  Stream 1 → HTML
  Stream 3 → JavaScript
  Stream 5 → Fonts
  Stream 7 → Conversation list
  Stream 9 → Streaming AI response
```

This is better than opening many separate TCP connections.

## Quick Comparison: SSE vs WebSocket

| Feature                        | Server-Sent Events            | WebSocket                     |
| ------------------------------ | ----------------------------- | ----------------------------- |
| Communication direction        | Mainly server → client        | Bidirectional                 |
| Transport                      | HTTP response stream          | WebSocket protocol            |
| Client sends data              | By normal HTTP request        | Over same socket              |
| Good for AI response streaming | Yes                           | Yes, but often unnecessary    |
| Built-in browser API           | EventSource                   | WebSocket                     |
| Can use fetch streaming        | Yes                           | No, WebSocket has its own API |
| Complexity                     | Lower                         | Higher                        |
| Best use case                  | Streaming updates from server | Real-time bidirectional apps  |

## Final Takeaway

ChatGPT-style streaming can be implemented using Server-Sent Events or fetch-based HTTP streaming.

The client sends a prompt to the backend. The backend keeps the response open and streams chunks of text as the model generates them. The frontend reads those chunks and renders them progressively.

This creates the live typing effect.

The lecture also shows that a real production application involves much more than just streaming: authentication, session handling, conversation IDs, message IDs, parent-child message links, moderation, title generation, feedback APIs, TLS, HTTP/2, and reverse proxy/CDN infrastructure all work together behind the scenes.
