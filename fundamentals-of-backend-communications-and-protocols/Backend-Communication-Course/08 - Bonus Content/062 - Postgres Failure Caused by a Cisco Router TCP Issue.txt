# 062 - Postgres Failure Caused by a Cisco Router TCP Issue

Section: 08 - Bonus Content
Duration: 2min

---

## 1. Summary

This lecture explains a real PostgreSQL connection failure caused by a router TCP idle timeout.

A PostgreSQL client connected to a PostgreSQL server and ran some queries. After that, the client spent more than 90 minutes processing data locally.

During those 90 minutes, no packets were exchanged between the client and the database server.

Between the client and the database, there was a Cisco router. The router tracked TCP connections in a local state table. However, because the connection was idle for more than the router's configured idle timeout, the router removed the connection state from its table.

The client and PostgreSQL server still believed the TCP connection was alive.

When the client finished local processing and tried to use the same PostgreSQL connection again, the router dropped the packet because it no longer recognized the connection.

The result was a PostgreSQL client connection failure.

The fix was to configure PostgreSQL TCP keepalive so that keepalive packets are sent before the router's idle timeout expires.

---

## 2. Key Concepts

### Stateful Router

Some routers track active TCP connections in a local state table.

This state helps the router make decisions about:

* routing,
* filtering,
* firewall behavior,
* invalid packet detection,
* connection tracking.

The router remembers that a TCP connection exists.

If it later receives packets that match the connection state, it allows them.

If it receives packets for a connection it no longer knows, it may drop them.

---

### Router State Table

A router state table stores information about active connections.

For a TCP connection, this may include the connection 4-tuple:

```text id="pt47dg"
Source IP
Source Port
Destination IP
Destination Port
```

The router uses this information to know which packets belong to an existing connection.

---

### TCP Idle Timeout

A TCP idle timeout is the amount of time a connection can remain inactive before the router removes its state.

A connection is considered idle when no packets are transmitted by either side.

The router assumes that if no data has been exchanged for a long time, the connection may be stale.

So it removes the state to save memory and avoid keeping dead connections forever.

---

### Cisco TCP Idle Time

The lecture mentions that Cisco routers may have a default TCP idle timeout of 1 hour.

This value can be configured with a setting such as:

```text id="dbi4gr"
tcp idle-time
```

If a TCP connection stays idle longer than this timeout, the router can remove the connection from its state table.

---

### PostgreSQL Long-Lived Connection

Database clients often keep PostgreSQL connections open for a long time.

This is normal.

A client may:

```text id="0kqte4"
connect to database
run query
process results locally
run another query later
```

The connection may stay open during the entire process.

However, if no packets are exchanged during the local processing period, intermediate routers may see the connection as idle.

---

### TCP Keepalive

TCP keepalive is a mechanism where one side sends small packets periodically to keep the connection active.

The purpose is to tell intermediate devices:

```text id="v1zue7"
This connection is still alive.
Do not remove its state.
```

Keepalive packets also help detect dead connections.

---

### PostgreSQL Keepalive Settings

PostgreSQL supports TCP keepalive configuration.

One important setting is:

```text id="91bvnb"
tcp_keepalives_idle
```

This controls how long the connection can be idle before keepalive messages are sent.

If the value is `0`, PostgreSQL uses the operating system default.

In the lecture's case, the system was on Windows, and the Windows default was around 2 hours.

That was too long because the Cisco router dropped idle connections after 1 hour.

---

## 3. Important Notes

### Both Client and Server May Think the Connection Is Alive

When the router removes connection state, the client and server may not immediately know.

The TCP connection still exists from their point of view.

No close packet was sent.

No reset packet was necessarily delivered.

So both sides may keep the socket open.

The failure only appears when one side tries to send data again.

---

### The Router Is in the Middle

The issue was not caused directly by PostgreSQL.

It was also not caused directly by the client application.

The failure happened because an intermediate router dropped connection state after an idle timeout.

This is a classic distributed systems problem:

```text id="h4z2ja"
Client thinks connection is alive.
Server thinks connection is alive.
Router forgot the connection.
Next packet gets dropped.
```

---

### Idle Does Not Mean Closed

A TCP connection can be open but idle.

Open means both endpoints still have the socket.

Idle means no packets are being exchanged.

Intermediate devices may treat long idle connections as stale.

---

### Keepalive Must Be Shorter Than the Router Timeout

If the router timeout is 60 minutes, keepalive should happen before 60 minutes.

For example:

```text id="loqch9"
Router idle timeout: 60 minutes
Postgres keepalive idle: 30 minutes
```

This causes PostgreSQL to send keepalive packets before the router removes state.

---

### OS Defaults May Not Match Network Defaults

PostgreSQL may use the operating system default if `tcp_keepalives_idle = 0`.

In the lecture, Windows default keepalive behavior was too slow for the Cisco router timeout.

So relying on OS defaults can be dangerous in networked database environments.

---

### Long Local Processing Can Break Idle Connections

The client ran queries, then processed data locally for over 90 minutes.

During processing, it did not communicate with PostgreSQL.

That idle period was enough for the router to forget the connection.

This can happen in applications that:

* stream data from database,
* process large result sets locally,
* perform long CPU tasks,
* wait for user input,
* pause between queries,
* keep database connections open for a long time.

---

### The Fix Is Simple Once the Cause Is Known

The fix was to configure PostgreSQL to send TCP keepalive packets more frequently.

Example:

```text id="j1px4v"
tcp_keepalives_idle < router_tcp_idle_timeout
```

If router timeout is 1 hour, configure PostgreSQL keepalive idle to less than 1 hour.


---

### Example 2: Why the Router Drops the Packet

Before timeout:

```text id="vecnjw"
Router state table:
Client:54321 → Postgres:5432 exists
```

After 60 minutes idle:

```text id="eu5hae"
Router state table:
No entry for Client:54321 → Postgres:5432
```

Client sends a packet later:

```text id="bfpn06"
Client:54321 → Postgres:5432
```

Router sees no matching state and drops it.

---

### Example 3: PostgreSQL Keepalive Fix

If the router idle timeout is 60 minutes, configure PostgreSQL keepalive to less than 60 minutes.

Example:

```conf id="tnvhrj"
tcp_keepalives_idle = 1800
```

This means PostgreSQL starts sending keepalive packets after 1800 seconds, or 30 minutes.

That keeps the router state alive before the 60-minute timeout.

---

### Example 4: Bad Configuration

```text id="podj5f"
Router idle timeout: 60 minutes
PostgreSQL keepalive idle: 120 minutes
```

Problem:

```text id="z9c7fc"
Router forgets the connection before PostgreSQL sends keepalive.
```

---

### Example 5: Good Configuration

```text id="f39q5m"
Router idle timeout: 60 minutes
PostgreSQL keepalive idle: 30 minutes
```

Result:

```text id="s905pz"
PostgreSQL sends keepalive before router timeout.
Router sees traffic.
Connection state remains active.
```

---

### Example 6: Application Pattern That Can Trigger This

```text id="6yl204"

process(rows) for 90 minutes locally

```

The second query may fail if the connection was idle too long and an intermediate router dropped the state.

## Final Takeaway

Long-lived database connections can fail if intermediate routers or firewalls drop idle TCP state.

The client and server may both believe the connection is still alive, but the router in the middle may have forgotten it.

The fix is to configure TCP keepalive so that packets are sent before the router's idle timeout expires.

For PostgreSQL, this means setting `tcp_keepalives_idle` to a value lower than the network device's TCP idle timeout.

In this case:

```text id="ohki83"
Cisco router idle timeout: 60 minutes
Windows default keepalive: 2 hours
Problem: keepalive happens too late

Fix:
PostgreSQL tcp_keepalives_idle < 60 minutes
```

Small keepalive packets can prevent a painful and confusing production connection failure.
