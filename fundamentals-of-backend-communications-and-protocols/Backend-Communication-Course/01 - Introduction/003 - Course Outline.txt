## 1. Summary

This lesson gives an overview of the full course structure. The instructor explains the major sections that will be covered, starting from backend communication design patterns and moving toward concrete protocols, HTTP/HTTPS configurations, backend execution patterns, proxies, and load balancers.

The course begins with the common ways clients communicate with backend systems, such as request/response, publish/subscribe, polling, long polling, push, server-sent events, synchronous communication, asynchronous communication, stateful systems, and stateless systems.

After that, the course moves into networking and protocol fundamentals. It introduces the OSI model, then explains concrete protocols such as UDP, TCP, TLS, HTTP/1.1, HTTP/2, HTTP/3, QUIC, gRPC, WebRTC, and WebSocket.

The later sections focus on how backend applications actually execute under the hood. This includes how the operating system and kernel accept connections, how raw bytes move from the kernel to the application process, how processes and threads work, and how different backend architectures affect latency and performance.

Finally, the course covers proxies and load balancers, including layer 4 proxies, layer 7 proxies, reverse proxies, API gateways, and content delivery networks.

### Backend Communication Design Patterns

This section explains the common ways clients and backend systems communicate. The instructor emphasizes that, although backend systems may seem complex, there are only a limited number of communication patterns that appear repeatedly in real-world applications.

Important patterns include:

* Request / Response

* Stateful Communication
* Stateless Communication
* Polling
* Push

* Server-Sent Events
* Sidecar Pattern
* Multiplexing and Demultiplexing

Each pattern has its own advantages, disadvantages, and suitable use cases.

---

### Synchronous vs Asynchronous Communication

Synchronous and asynchronous communication are core ideas in backend engineering.

In synchronous communication, the client usually waits for the backend to finish processing and return a response.

In asynchronous communication, the backend may accept the request and process it later, often using queues, workers, or event-driven systems.

These concepts appear frequently throughout backend design, scalability, and system architecture.

---

### Stateful vs Stateless Communication

A stateful system remembers information about previous interactions. A stateless system treats each request independently.

The course discusses:

* What makes a protocol stateful or stateless
* Why stateful systems can be useful
* Why stateless systems are easier to scale
* The trade-offs between both approaches

---

### Sidecar Pattern

The sidecar pattern is common in microservices and service mesh architectures. It usually involves running a helper proxy beside the main application.

The sidecar can handle tasks such as:

* Networking
* Routing
* Security
* Observability
* Service-to-service communication

Although it is based on a simple proxy idea, it becomes very powerful in distributed systems.

---

### Multiplexing and Demultiplexing

Multiplexing is a low-level networking concept, but it appears frequently in backend systems.

It can be seen in:

* HTTP/2
* QUIC
* Multipath TCP
* Database connections
* Browser connection management
* Proxies and connection pooling

Multiplexing allows multiple streams or requests to share the same underlying connection.

## 3. Important Notes

* The course starts from high-level backend communication patterns and gradually moves into low-level networking and execution details.
* Most backend systems can be understood as combinations of common communication patterns.
* Each communication pattern has trade-offs. No single pattern is always the best.
* Protocols are critical because they define how systems communicate.
* The OSI model is important because it provides a common language for discussing networking layers.
* HTTP and HTTPS are central to modern web systems, so they deserve special attention.
* Backend performance is heavily affected by execution models, processes, threads, kernel behavior, and connection handling.
* The operating system and kernel perform many tasks that backend engineers often do not see.
* Proxies and load balancers are fundamental parts of backend architecture.
* Understanding backend internals helps engineers debug, optimize, and design better systems.

## 4. Examples

### Example 1: Request / Response

A user opens a website and the browser sends a request to the backend:

```text id="5g1fxp"
Client → Request → Backend
Client ← Response ← Backend
```

This is the classic request/response model.

It is simple and widely used in APIs, web applications, and HTTP services.

---

### Example 2: Publish / Subscribe

In a publish/subscribe model, one service publishes messages to a broker, and other services subscribe to receive those messages.

```text id="93wxrp"
Producer → Message Broker → Subscribers
```

Examples of tools that use this style include Kafka and RabbitMQ.

This pattern is useful for event-driven systems and asynchronous processing.

---

### Example 3: Polling vs Push

In polling, the client repeatedly asks the backend for updates.

```text id="wwkm6h"
Client → Any new data?
Backend → No

Client → Any new data?
Backend → Yes
```

In push, the backend sends data to the client when new information is available.

```text id="sed7sl"
Backend → New update → Client
```

Polling is simple, but it can waste resources. Push is more efficient for real-time updates, but it requires a different communication model.

---

### Example 4: HTTP/1.1 vs HTTP/2

HTTP/1.1 often uses multiple connections to handle multiple requests.

HTTP/2 supports multiplexing, allowing multiple streams over the same connection.

```text id="tst7vg"
HTTP/1.1:
Request 1 → Connection 1
Request 2 → Connection 2

HTTP/2:
Request 1, Request 2, Request 3 → Same Connection
```

This can improve performance and reduce connection overhead.

---

### Example 5: Backend Execution

When a request reaches the backend, the application does not receive a clean “request object” immediately.

Many steps happen first:

```text id="vsr9fl"
Network → Kernel → Socket → Process → Runtime → Framework → Application Code
```

The framework hides most of this complexity, but understanding it helps engineers debug deeper issues.

---

### Example 6: Reverse Proxy

A reverse proxy sits in front of backend services and forwards requests to the correct service.

```text id="g5ep7y"
Client → Reverse Proxy → Backend Service
```

A reverse proxy can handle:

* Routing
* Load balancing
* TLS termination
* Compression
* Caching
* Protocol translation

Examples include Nginx, Envoy, HAProxy, and API gateways.
