# 030 - WebRTC

**Section:** 03 - Protocols  
**Duration:** 1hr 8min  

---

## 1. Summary

WebRTC stands for:

```text
Web Real-Time Communication
```

WebRTC is a free and open-source technology that allows browsers and mobile applications to communicate in real time.

The main goal of WebRTC is to provide low-latency communication between peers.

It is commonly used for:

- Video calls
- Voice calls
- Screen sharing
- Peer-to-peer data transfer
- Real-time chat
- Live collaboration
- Low-latency media communication

The ideal WebRTC path is peer-to-peer.

```text
Peer A ⇄ Peer B
```

This avoids sending all traffic through a central application server.

Peer-to-peer communication can reduce latency because data travels through the shortest practical path between two peers.

However, WebRTC is complex because most users are behind NAT, firewalls, routers, and private networks.

To make WebRTC work, several technologies are involved:

```text
NAT
STUN
TURN
ICE
SDP
Signaling
RTCPeerConnection
DataChannel
MediaStream
```

---

## 2. Key Concepts

## 2.1 Why WebRTC Exists

Before WebRTC, real-time browser communication was difficult.

If you wanted audio/video communication, you often needed:

- Native apps
- Browser plugins
- Central media servers
- Custom protocols
- Complex setup

WebRTC gives browsers built-in APIs for real-time communication.

The browser can access:

- Camera
- Microphone
- Screen
- Network candidates
- Peer connection
- Data channel

This makes it possible to build real-time apps directly in the browser.

---

## 2.2 WebRTC Is Designed for Low Latency

For video and audio, latency matters a lot.

If every packet must go through a central server, latency increases.

Example with relay server:

```text
Peer A → Server → Peer B
```

Better path:

```text
Peer A → Peer B
```

Peer-to-peer is usually faster because there is no application server in the middle processing and forwarding all media.

---

## 2.3 WebRTC Is Not Just One Protocol

WebRTC is not a single simple protocol.

It is a collection of APIs and protocols working together.

Important pieces:

| Component | Purpose |
|---|---|
| NAT | Private network translation |
| STUN | Discover public IP/port |
| TURN | Relay traffic when direct connection fails |
| ICE | Find the best connection path |
| SDP | Describe session information |
| Signaling | Exchange SDP and candidates |
| RTCPeerConnection | Browser API for peer connection |
| DataChannel | Send arbitrary data |
| MediaStream | Send audio/video |

---

## 2.4 Peer-to-Peer Communication

The ideal WebRTC connection is peer-to-peer.

```text
Browser A ⇄ Browser B
```

But peers often do not know how to reach each other.

Why?

Because they may be behind:

- NAT
- Home router
- Corporate firewall
- Mobile carrier NAT
- Symmetric NAT
- Private IP address

So WebRTC first needs to discover possible ways for peers to connect.

---

## 2.5 Signaling

WebRTC does not define how signaling must happen.

Signaling means exchanging connection information before the peer-to-peer connection can start.

This information includes:

- SDP offer
- SDP answer
- ICE candidates
- Media capabilities
- Security parameters
- Network information

Signaling can happen through:

- WebSocket
- HTTP
- Socket.IO
- QR code
- Copy/paste
- Messaging app
- Any custom channel

Important idea:

```text
WebRTC gives you peer-to-peer communication,
but you still need some way to exchange setup information first.
```

---

## 2.6 Why Signaling Feels Strange

WebRTC is used to help two peers communicate.

But before they can communicate, they must exchange setup information somehow.

This can feel strange:

```text
I need communication to establish communication.
```

The reason is that signaling only sends small setup metadata.

After that, WebRTC can create a better direct media/data path.

Example:

```text
WebSocket used for signaling
WebRTC used for media/data
```

---

## 3. NAT

## 3.1 What Is NAT?

NAT stands for:

```text
Network Address Translation
```

Most home users are behind NAT.

Example:

```text
Laptop private IP: 192.168.1.22
Router public IP: 5.5.5.5
```

The laptop does not directly expose a public IP address.

The router owns the public IP address.

When the laptop sends traffic to the internet, the router rewrites the source IP and port.

---

## 3.2 Why NAT Exists

Private IP addresses are not routable on the public internet.

Examples of private IP ranges:

```text
192.168.x.x
10.x.x.x
172.16.x.x - 172.31.x.x
```

If a device with private IP `192.168.1.22` sends a packet to the internet, the outside server cannot reply directly to that private address.

The router must translate it.

---

## 3.3 NAT Example

Internal machine:

```text
192.168.1.22:8992
```

Router public mapping:

```text
5.5.5.5:3333
```

Destination server:

```text
4.4.4.4:80
```

The router creates a NAT table entry:

```text
Internal: 192.168.1.22:8992
External: 5.5.5.5:3333
Destination: 4.4.4.4:80
```

When the response comes back to:

```text
5.5.5.5:3333
```

The router knows it should forward it to:

```text
192.168.1.22:8992
```

---

## 3.4 NAT Table

A NAT table maps internal connections to external public mappings.

Example:

```text
192.168.1.22:8992 → 5.5.5.5:3333
```

This table lets the router know how to send responses back to the correct internal device.

---

## 4. NAT Types

Different NAT types behave differently.

This matters a lot for WebRTC.

Common NAT types:

```text
Full Cone NAT
Address-Restricted Cone NAT
Port-Restricted Cone NAT
Symmetric NAT
```

WebRTC works more easily with the first three.

Symmetric NAT is harder and often requires TURN.

---

## 4.1 Full Cone NAT

Full Cone NAT is the most permissive.

Once an internal address is mapped to an external IP/port, any external host can send packets to that external mapping.

Example mapping:

```text
192.168.1.22:8992 → 5.5.5.5:3333
```

Any external peer can send to:

```text
5.5.5.5:3333
```

The router forwards it to:

```text
192.168.1.22:8992
```

This is good for peer-to-peer communication.

---

## 4.2 Address-Restricted Cone NAT

Address-Restricted Cone NAT only allows packets from external IP addresses that the internal client has contacted before.

Example:

```text
Internal client contacted 4.4.4.4
```

Then packets from `4.4.4.4` can come back.

But packets from a new unknown IP may be blocked.

This is stricter than Full Cone NAT.

---

## 4.3 Port-Restricted Cone NAT

Port-Restricted Cone NAT checks both IP address and port.

The external sender must match an IP and port that the internal client previously contacted.

Example allowed:

```text
Previously contacted: 4.4.4.4:80
Incoming from:        4.4.4.4:80
```

Example blocked:

```text
Previously contacted: 4.4.4.4:80
Incoming from:        4.4.4.4:443
```

This is stricter than Address-Restricted Cone NAT.

---

## 4.4 Symmetric NAT

Symmetric NAT is the most restrictive.

The mapping is specific to the destination IP and port.

Example:

```text
Internal client → STUN server
Mapping created only for that STUN server
```

The same public IP/port mapping may not work for another peer.

This is a problem for WebRTC.

Why?

Because the public mapping discovered through STUN may only work with the STUN server, not with the remote peer.

When Symmetric NAT is involved, direct peer-to-peer connection may fail.

In that case, WebRTC often needs TURN.

---

## 5. STUN

## 5.1 What Is STUN?

STUN stands for:

```text
Session Traversal Utilities for NAT
```

STUN helps a peer discover its public IP address and port as seen by the outside world.

A STUN server answers the question:

```text
What is my public IP and port?
```

---

## 5.2 Why STUN Is Needed

A browser behind NAT may only know its private IP address.

Example:

```text
192.168.1.22
```

But the remote peer cannot connect to this private IP address.

The browser needs to know its public mapping.

Example:

```text
5.5.5.5:3333
```

STUN helps discover that mapping.

---

## 5.3 STUN Flow

Simplified STUN flow:

```text
1. Browser sends request to STUN server.
2. NAT router creates a public mapping.
3. STUN server sees the public IP/port.
4. STUN server sends that public IP/port back.
5. Browser learns its public presence.
```

Example:

```text
Browser private: 192.168.1.22:8992
Router public:   5.5.5.5:3333
STUN server says: You are 5.5.5.5:3333
```

---

## 5.4 STUN Is Cheap

STUN is lightweight.

It does not relay media.

It only helps discover public mapping information.

That is why public STUN servers are common.

Common STUN port:

```text
3478
```

STUN over TLS often uses:

```text
5349
```

---

## 5.5 STUN Limitation

STUN does not work well with Symmetric NAT.

If the NAT mapping is only valid for the STUN server, another peer cannot use that mapping to connect.

In that case, TURN is needed.

---

## 6. TURN

## 6.1 What Is TURN?

TURN stands for:

```text
Traversal Using Relays around NAT
```

TURN is used when direct peer-to-peer communication fails.

TURN relays traffic between peers.

Flow:

```text
Peer A → TURN Server → Peer B
Peer B → TURN Server → Peer A
```

---

## 6.2 Why TURN Is Needed

TURN is needed when NAT or firewall rules prevent direct peer-to-peer communication.

Common cases:

- Symmetric NAT
- Strict corporate firewall
- UDP blocked
- Carrier-grade NAT
- Direct ICE candidates fail

TURN acts as a fallback.

---

## 6.3 TURN Is Expensive

TURN is more expensive than STUN because it relays actual traffic.

For video and audio, this can mean a lot of bandwidth.

Example:

```text
Video stream from A to TURN
TURN forwards video stream to B
```

The TURN server must handle the media traffic.

This costs:

- Bandwidth
- CPU
- Public IP
- Infrastructure
- Operations

Because of this, TURN servers are usually not freely provided at scale.

---

## 6.4 TURN vs STUN

| Feature | STUN | TURN |
|---|---|---|
| Purpose | Discover public IP/port | Relay traffic |
| Cost | Cheap | Expensive |
| Media path | Peer-to-peer if possible | Through relay |
| Used when | NAT traversal works | Direct connection fails |
| Bandwidth load | Low | High |

---

## 7. ICE

## 7.1 What Is ICE?

ICE stands for:

```text
Interactive Connectivity Establishment
```

ICE is the process WebRTC uses to find the best possible connection path between peers.

ICE collects candidates.

A candidate is a possible way to connect to a peer.

---

## 7.2 ICE Candidates

ICE candidates can include:

```text
Host candidates
Server-reflexive candidates
Relay candidates
```

---

## 7.3 Host Candidate

A host candidate is a local network address.

Example:

```text
192.168.1.22:5000
```

This may work if both peers are on the same local network.

---

## 7.4 Server-Reflexive Candidate

A server-reflexive candidate is discovered through STUN.

Example:

```text
5.5.5.5:3333
```

This is the public IP/port mapping created by NAT.

---

## 7.5 Relay Candidate

A relay candidate is provided by TURN.

Example:

```text
TURN server IP/port
```

If direct peer-to-peer paths fail, traffic may go through this relay.

---

## 7.6 ICE Candidate Collection

ICE candidate collection can take time.

The browser may discover candidates gradually.

This is called:

```text
ICE trickling
```

As candidates are discovered, they can be sent to the remote peer.

---

## 7.7 Choosing the Best Path

ICE tries different candidates and chooses the best working path.

Preferred order is usually:

```text
1. Direct local connection
2. Direct public/STUN-based connection
3. TURN relay
```

The goal is to use the lowest-latency path that actually works.

---

## 8. SDP

## 8.1 What Is SDP?

SDP stands for:

```text
Session Description Protocol
```

SDP is a text format that describes a WebRTC session.

It contains information such as:

- ICE candidates
- Network options
- Media options
- Codecs
- Security parameters
- Data channel information
- Audio/video capabilities

---

## 8.2 SDP Is the Thing You Signal

The main job of signaling is to send SDP between peers.

Example:

```text
Peer A creates offer SDP.
Peer A sends offer SDP to Peer B.
Peer B creates answer SDP.
Peer B sends answer SDP back to Peer A.
```

After both sides set local and remote descriptions, the connection can be established.

---

## 8.3 Offer and Answer

WebRTC uses an offer/answer model.

Peer A creates an offer.

```text
Offer = Peer A's SDP
```

Peer B receives the offer and creates an answer.

```text
Answer = Peer B's SDP
```

Then Peer A receives the answer.

After this exchange, peers have enough information to connect.

---

## 8.4 Local Description and Remote Description

Each peer has:

```text
Local description
Remote description
```

The local description is the peer's own SDP.

The remote description is the other peer's SDP.

Example for Peer A:

```text
Peer A local description  = A's offer
Peer A remote description = B's answer
```

Example for Peer B:

```text
Peer B remote description = A's offer
Peer B local description  = B's answer
```

---

## 8.5 SDP Example Shape

SDP is usually a long string.

It may contain lines like:

```text
v=0
o=...
s=...
t=0 0
a=ice-ufrag:...
a=ice-pwd:...
a=fingerprint:...
a=candidate:...
m=audio ...
m=video ...
```

You usually do not write SDP manually.

The browser generates it.

---

## 9. Signaling

## 9.1 What Is Signaling?

Signaling is the process of exchanging setup information between peers.

WebRTC does not define the signaling protocol.

You decide how to send the SDP and ICE candidates.

Possible signaling channels:

```text
WebSocket
HTTP
Socket.IO
QR code
Copy/paste
WhatsApp
Custom server
```

---

## 9.2 Common Signaling Flow

Simplified flow:

```text
1. Peer A creates RTCPeerConnection.
2. Peer A creates offer.
3. Peer A sets offer as local description.
4. Peer A sends offer to Peer B through signaling.
5. Peer B sets offer as remote description.
6. Peer B creates answer.
7. Peer B sets answer as local description.
8. Peer B sends answer to Peer A through signaling.
9. Peer A sets answer as remote description.
10. ICE connects peers using best candidate path.
```

---

## 9.3 Why WebSocket Is Often Used for Signaling

WebSocket is commonly used because signaling is interactive.

Peers may need to exchange:

- Offer
- Answer
- New ICE candidates
- Connection state updates

WebSocket gives a convenient bidirectional channel for that setup phase.

But signaling can be done using any method.

---

## 10. WebRTC Demo Concept

## 10.1 Goal

Create a simple data channel between two browser tabs.

No server-side application logic is needed for the basic demo if you manually copy/paste SDP between tabs.

The demo has:

```text
Peer A
Peer B
DataChannel
Manual signaling
```

---

## 10.2 Peer A Creates Offer

In Browser A:

```js
const peerA = new RTCPeerConnection();

const channel = peerA.createDataChannel("chat");

channel.onopen = () => {
  console.log("Peer A data channel open");
};

channel.onmessage = (event) => {
  console.log("Peer A received:", event.data);
};

peerA.onicecandidate = () => {
  console.log("Peer A SDP:");
  console.log(JSON.stringify(peerA.localDescription));
};

const offer = await peerA.createOffer();
await peerA.setLocalDescription(offer);
```

After ICE gathering, copy Peer A's local description and send it to Peer B.

This copied object is the offer.

---

## 10.3 Peer B Receives Offer and Creates Answer

In Browser B:

```js
const offer = /* paste Peer A offer here */;

const peerB = new RTCPeerConnection();

peerB.ondatachannel = (event) => {
  const channel = event.channel;

  channel.onopen = () => {
    console.log("Peer B data channel open");
  };

  channel.onmessage = (event) => {
    console.log("Peer B received:", event.data);
  };

  peerB.channel = channel;
};

peerB.onicecandidate = () => {
  console.log("Peer B SDP:");
  console.log(JSON.stringify(peerB.localDescription));
};

await peerB.setRemoteDescription(offer);

const answer = await peerB.createAnswer();
await peerB.setLocalDescription(answer);
```

After ICE gathering, copy Peer B's local description and send it back to Peer A.

This copied object is the answer.

---

## 10.4 Peer A Receives Answer

In Browser A:

```js
const answer = /* paste Peer B answer here */;

await peerA.setRemoteDescription(answer);
```

After this, the data channel should open.

---

## 10.5 Send Messages

From Peer A:

```js
channel.send("Hello Peer B");
```

From Peer B:

```js
peerB.channel.send("Hello Peer A");
```

This demonstrates WebRTC peer-to-peer data communication.

## 11. Media API

## 11.1 getUserMedia

To send audio or video, use:

```js
navigator.mediaDevices.getUserMedia()
```

Example:

```js
const stream = await navigator.mediaDevices.getUserMedia({
  audio: true,
  video: true,
});
```

This asks the browser for microphone and camera access.

---

## 11.2 Add Media Tracks

After getting a media stream, add its tracks to the peer connection:

```js
for (const track of stream.getTracks()) {
  peerConnection.addTrack(track, stream);
}
```

The rest of the WebRTC process is similar:

```text
Create offer
Signal offer
Create answer
Signal answer
ICE connects peers
Media flows
```

## 12. Custom STUN and TURN Servers

## 12.1 ICE Server Configuration

You can configure STUN and TURN servers when creating an `RTCPeerConnection`.

Example:

```js
const peerConnection = new RTCPeerConnection({
  iceServers: [
    {
      urls: "stun:stun.l.google.com:19302",
    },
    {
      urls: "turn:turn.example.com:3478",
      username: "user",
      credential: "password",
    },
  ],
});
```

---

## 12.2 Public STUN Servers

Public STUN servers are common because STUN is cheap to operate.

Example:

```text
stun:stun.l.google.com:19302
```

Public STUN is useful for testing and simple applications.

---

## 12.3 TURN Servers Need Credentials

TURN servers relay traffic.

Because relay bandwidth is expensive, TURN servers usually require credentials.

Example:

```js
{
  urls: "turn:turn.example.com:3478",
  username: "user",
  credential: "password"
}
```

---

## 12.4 Running Your Own TURN Server

A common open-source TURN/STUN server is:

```text
coturn
```

With your own TURN server, you can control:

- Authentication
- Bandwidth
- Regions
- Logging
- Availability
- Security policies

## 13.1 Peer-to-Peer

WebRTC can create direct peer-to-peer connections.

This can reduce latency.

```text
Peer A ⇄ Peer B
```

---

## 13.2 Low Latency

WebRTC is designed for real-time communication.

It is useful for:

- Audio
- Video

WebRTC works directly in modern browsers.

You do not need plugins.

Important APIs:

```text
RTCPeerConnection

```

---

## 13.4 Standardized

Because WebRTC is standardized and implemented in browsers, developers can build real-time apps across platforms.

- Audio

```text
RTCDataChannel
```

```text
MediaStream tracks
```

## 14. Cons of WebRTC

## 14.1 Complexity

WebRTC has many moving parts.

You need to understand:

- NAT
- STUN
- TURN
- ICE
- SDP
- Signaling
- Media tracks
- Data channels
- Browser permissions
- Connection states

This makes WebRTC harder than simple HTTP or WebSocket.

---

## 14.2 Peer-to-Peer Does Not Always Work

Some networks prevent direct peer-to-peer communication.

Examples:

- Symmetric NAT
- Strict firewalls
- UDP blocked
- Corporate networks
- Mobile carrier NAT

When peer-to-peer fails, TURN is needed.

---

## 14.3 TURN Is Expensive

TURN relays media traffic.

For video calls, this can be expensive.

A TURN server must handle large bandwidth.

---

## 14.4 Multi-Participant Mesh Does Not Scale Well

If many users connect peer-to-peer with each other, the number of connections grows quickly.

Example with 100 participants:

```text
Every participant connects to many other participants.
```

This becomes a mesh.

It can overload clients and networks.

For large calls, systems often use media servers such as:

- SFU
- MCU
- Custom relay servers

---

## 14.5 Signaling Is Not Provided

WebRTC does not provide signaling.

You must build or choose your own signaling mechanism.

Common choice:

```text
WebSocket signaling server
```

---

## 14.6 Debugging Can Be Hard

WebRTC failures can come from many places:

- NAT type
- Firewall
- TURN credentials
- ICE candidate failure
- SDP mismatch
- Browser permissions
- Media device problems
- Network changes

This makes debugging challenging.

---

## 14.7 Privacy Concerns

WebRTC may expose network-related information through ICE candidates.

Modern browsers have improved privacy protections, but engineers should still understand what information is exposed and how candidates are handled.

---

## 15. WebRTC vs WebSocket

| Feature | WebRTC | WebSocket |
|---|---|---|
| Main purpose | Real-time peer-to-peer media/data | Client-server bidirectional communication |
| Topology | Usually peer-to-peer | Client-server |
| Media support | Built in | Not built in |
| NAT traversal | Built in using ICE/STUN/TURN | Not peer-to-peer focused |
| Signaling needed | Yes | No, WebSocket itself is the connection |
| Best for | Video/audio/data peer communication | Chat, notifications, backend push |
| Complexity | Higher | Lower |

---

## 16. WebRTC Connection Flow

Full high-level flow:

```text
1. Peer A creates RTCPeerConnection.
2. Peer A collects ICE candidates.
3. Peer A creates SDP offer.
4. Peer A sends offer to Peer B through signaling.
5. Peer B creates RTCPeerConnection.
6. Peer B sets offer as remote description.
7. Peer B collects ICE candidates.
8. Peer B creates SDP answer.
9. Peer B sends answer to Peer A through signaling.
10. Peer A sets answer as remote description.
11. ICE checks candidate pairs.
12. Best connection path is selected.
13. Media/data starts flowing.
```

---

## 17. Important Notes

## 17.1 WebRTC Needs Signaling

WebRTC does not magically connect peers.

You need a signaling mechanism to exchange SDP and ICE candidates.

---

## 17.2 SDP Is Critical

SDP describes the session.

It contains the connection, media, codec, security, and ICE information needed to establish communication.

---

## 17.3 ICE Finds the Best Path

ICE collects possible connection paths and tests them.

The best working path is selected.

---

## 17.4 STUN Discovers Public Mapping

STUN tells a peer what its public IP and port look like from the outside.

---

## 17.5 TURN Relays When Direct Connection Fails

TURN is the fallback when direct peer-to-peer communication cannot be established.

---

## 17.6 Peer-to-Peer Is Not Guaranteed

WebRTC tries to create a peer-to-peer path, but it may fall back to TURN.

---

## 17.7 TURN Can Become the Real Architecture

If many users cannot connect directly, your TURN infrastructure becomes very important.

This affects cost, scaling, and reliability.

---

## 17.8 Multi-Party Calls Need Extra Architecture

For many participants, pure peer-to-peer mesh is often not enough.

Use SFU, MCU, or another media server architecture.

---

What does STUN do?

**Answer:**

STUN helps a peer discover its public IP address and port.

---

## Question 9

What does TURN do?

**Answer:**

TURN relays traffic when direct peer-to-peer communication fails.

---

## Question 10

Why is TURN more expensive than STUN?

**Answer:**

Because TURN carries actual media/data traffic, while STUN only helps discover public mapping information.

What does ICE stand for?

**Answer:**

ICE stands for **Interactive Connectivity Establishment**.

## Question 12

What is an ICE candidate?

**Answer:**

An ICE candidate is a possible network path that can be used to connect to a peer.

## Question 13

What are the three common ICE candidate types?

```text
Relay
```

What does SDP stand for?

## Question 15

What does SDP contain?

**Answer:**

SDP contains session information such as ICE candidates, media information, codecs, and security parameters.

## Question 16

What is an offer?

**Answer:**

An offer is SDP created by the peer that starts the WebRTC negotiation.

---

## Question 17

What is an answer?

**Answer:**

An answer is SDP created by the receiving peer in response to the offer.

## Question 18

What is RTCPeerConnection?

**Answer:**

`RTCPeerConnection` is the main browser API used to create and manage a WebRTC peer connection.

## Question 19

What is RTCDataChannel?

**Answer:**

`RTCDataChannel` is a WebRTC API used to send arbitrary data between peers.

## Question 20

Why does peer-to-peer not scale well for many participants?

**Answer:**

Because every participant may need connections to many other participants, creating a large mesh of connections.

---

## Question 21

What should you use for large multi-party video calls?

**Answer:**

Use a media server architecture such as SFU, MCU, or a custom relay system.

---

## Question 22

When is WebRTC better than WebSocket?

**Answer:**

WebRTC is better when you need low-latency peer-to-peer media or data communication.

---

## Question 23

When is WebSocket better than WebRTC?

**Answer:**

WebSocket is better for simpler client-server real-time communication, such as notifications or chat through a backend server.

## 20. Final Takeaway

WebRTC is a powerful technology for real-time communication in browsers and mobile applications.

Its goal is:

```text
Create the lowest-latency practical path between peers.
```

The most important WebRTC concepts are:

```text
NAT
STUN
TURN
ICE
ICE candidates
SDP
Offer
Answer
Signaling
RTCPeerConnection
RTCDataChannel
MediaStream
```

The simplified WebRTC flow is:

```text
1. Create peer connection.
2. Collect ICE candidates.
3. Create SDP offer.
4. Signal offer to remote peer.
5. Remote peer creates SDP answer.
6. Signal answer back.
7. ICE selects the best path.
8. Peers communicate directly if possible.
9. If direct path fails, TURN relays traffic.
```

WebRTC is excellent for:

```text
Video calls
Voice calls
Screen sharing
Peer-to-peer data
Low-latency communication
```

But it is complex.

The hardest parts are:

```text
NAT traversal
TURN cost
Signaling
Debugging
Multi-party scaling
Network variability
```

As a backend engineer, the key lesson is:

```text
WebRTC is not just an API.
It is a system of protocols working together.
```

Use WebRTC when peer-to-peer, low-latency communication is truly needed.

Use simpler tools like WebSocket or SSE when a normal client-server real-time model is enough.
