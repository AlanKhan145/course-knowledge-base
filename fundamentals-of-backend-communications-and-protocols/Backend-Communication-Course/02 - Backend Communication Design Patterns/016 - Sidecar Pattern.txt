# 016 - Sidecar Pattern

**Section:** 02 - Backend Communication Design Patterns
**Duration:** 22min

---

## 1. Summary

This lesson explains the **Sidecar Pattern**, a design pattern commonly used in microservices and service mesh architectures.

The main problem this pattern solves is protocol and communication complexity. Every protocol requires a library that knows how to speak that protocol. For example, HTTP/1.1, HTTP/2, HTTP/3, gRPC, TLS, SOAP, TCP, and UDP all require specific client and server logic.

As protocols become more complex, application libraries become larger and harder to upgrade. Changing a communication library can require code changes, retesting, compatibility checks, and redeployment.

The sidecar pattern solves this by moving communication responsibility out of the application and into a separate local proxy. The application talks to the sidecar using a simple protocol, often HTTP/1.1, while the sidecar handles complex networking tasks such as protocol upgrades, TLS, retries, tracing, service discovery, load balancing, and monitoring.

This pattern is heavily used in service meshes such as Envoy, Istio, and Linkerd.

---

## 2. Key Concepts

### Sidecar Pattern

The sidecar pattern places a helper process or proxy beside the main application.

```text id="fuz3ff"
Application → Sidecar Proxy → Network
```

The sidecar runs close to the application, usually on the same machine, container group, or Kubernetes pod.

---

### Why It Is Called Sidecar

It is called “sidecar” because the proxy runs next to the application, like a motorcycle sidecar attached beside the main vehicle.

The application remains the main service, while the sidecar assists with communication.

---

### Protocol Complexity

Every protocol requires code that understands how to write and read that protocol.

Examples:

* HTTP/1.1
* HTTP/2
* HTTP/3
* TLS
* gRPC
* SOAP
* TCP
* UDP

Even simple requests require protocol-specific formatting, parsing, headers, framing, encryption, and response handling.

---

### Protocol Library

A protocol library handles the details of a protocol.

For example, an HTTP library knows how to:

* Create HTTP requests
* Add headers
* Write to sockets
* Parse HTTP responses
* Handle connection reuse
* Deal with protocol versions

Applications depend heavily on these libraries.

---

### Thick Client

A thick client contains a lot of communication logic inside the application itself.

For example, a backend service may directly include libraries for:

* HTTP
* TLS
* gRPC
* Retries
* Timeouts
* Circuit breaking
* Service discovery
* Tracing

This can make the application harder to maintain.

---

### Library Upgrade Problem

Changing a communication library can be difficult because it may require:

* Code changes
* Retesting
* Compatibility checks
* Security review
* Redeployment
* Handling breaking changes

If many microservices use the same library, upgrading every service becomes painful.

---

### Same-Language Limitation

Most protocol libraries are language-specific.

For example:

```text id="3b3j55"
Python app → Python HTTP library
Node.js app → Node.js WebSocket library
Java app → Java gRPC library
```

This can limit teams if all services must use the same communication library.

---

### Delegating Communication

The sidecar pattern delegates communication responsibility to a separate proxy.

Instead of the application handling complex protocols directly, the sidecar handles them.

```text id="fbqfwg"
Application → Simple local request → Sidecar
Sidecar → Advanced protocol → Remote sidecar/server
```

---

### Local Proxy

The sidecar often runs locally and communicates with the application through loopback.

```text id="u79r04"
Application → localhost → Sidecar Proxy
```

Using loopback makes communication predictable because the local address does not change.

---

### Sidecar Proxy to Sidecar Proxy

In a service mesh, each service may have its own sidecar proxy.

```text id="elxopp"
Service A → Sidecar A → Network → Sidecar B → Service B
```

The application code stays simple, while the sidecars handle the network.

---

### Service Mesh

A service mesh is an infrastructure layer that manages service-to-service communication.

Common service mesh tools include:

* Istio
* Linkerd
* Envoy-based systems

Service mesh systems commonly use the sidecar pattern.

---

### Layer 7 Proxy

A sidecar proxy is usually a **Layer 7 proxy**, meaning it understands application-level protocols such as HTTP or gRPC.

Because it operates at Layer 7, it can inspect requests, headers, routes, methods, and metadata.

This allows it to support:

* Routing
* Tracing
* Retries
* Authentication
* Authorization
* Observability

---

### ALPN

ALPN stands for **Application-Layer Protocol Negotiation**.

It is a TLS extension that allows client and server to negotiate which application protocol to use, such as HTTP/1.1 or HTTP/2.

Example:

```text id="tqwshe"
Client offers: HTTP/2, HTTP/1.1
Server chooses: HTTP/2
```

The sidecar can handle this negotiation instead of the application.

---

### Protocol Upgrade

A sidecar can allow services to use newer protocols without changing application code.

Example:

```text id="bx941j"
Application speaks HTTP/1.1 locally
Sidecar speaks HTTP/2 or HTTP/3 over the network
```

This makes protocol upgrades easier.

---

### Security

Sidecars can centralize security features.

For example:

* TLS
* mTLS
* Certificate rotation
* Cipher configuration
* Authentication
* Authorization

If a TLS library or cipher becomes unsafe, the proxy configuration can be updated without changing every application.

---

### Tracing and Monitoring

Sidecars can automatically add tracing and monitoring.

They can attach a trace ID to requests and follow the request across multiple services.

```text id="gx2v9z"
Service A → Service B → Service C → Service D
```

The sidecars can track the whole path.

---

### Service Discovery

Sidecars can handle service discovery.

Instead of each application knowing where every service lives, the sidecar can find the destination service.

```text id="uxn0qj"
Application calls: user-service
Sidecar resolves actual location
Sidecar routes request
```

---

### Circuit Breaking and Retries

Sidecars can handle networking logic such as:

* Retries
* Timeouts
* Circuit breaking
* Load balancing
* Connection pooling
* Rate limiting

This keeps application code simpler.

---

## 3. Important Notes

* The sidecar pattern moves networking complexity out of the application.
* The application talks to a local proxy.
* The sidecar proxy handles advanced protocol and networking behavior.
* Sidecars are common in microservices and service mesh architectures.
* A sidecar can allow protocol upgrades without changing application code.
* A sidecar can make a system language-agnostic.
* Sidecars help polyglot microservice architectures.
* The sidecar can handle TLS, mTLS, retries, tracing, monitoring, service discovery, and caching.
* Sidecars are usually Layer 7 proxies.
* Because sidecars inspect application traffic, they must be trusted.
* Sidecars can decrypt, inspect, rewrite, and re-encrypt requests.
* Sidecars add extra hops to every request.
* Extra hops add latency.
* Sidecars increase system complexity.
* Debugging service mesh problems can be difficult.
* The sidecar pattern is powerful but not free.
* It should be used when the benefits justify the operational complexity.

## 4. Examples

### Example 1: Without Sidecar

Each application directly handles networking.

```text id="wvrxwg"
Service A → gRPC library → TLS library → Network → Service B
```

Problems:

* Application must include complex libraries.
* Each language needs compatible libraries.
* Upgrades require application changes.
* Security fixes must be rolled out across many services.

---

### Example 2: With Sidecar

The application sends a simple request to the local sidecar.

```text id="99ftcr"
Service A → Sidecar A → Network → Sidecar B → Service B
```

The sidecars handle communication complexity.

---

### Example 3: Local HTTP/1.1, Network HTTP/2

```text id="xcu9c1"
Application → HTTP/1.1 → Local Sidecar
Local Sidecar → HTTP/2 + TLS → Remote Sidecar
Remote Sidecar → HTTP/1.1 → Remote Application
```

The application does not need to understand HTTP/2.

---

### Example 4: Protocol Upgrade

Suppose the application only knows HTTP/1.1.

Later, the infrastructure team wants to use HTTP/3.

With sidecars:

```text id="hbzj53"
Application code: unchanged
Sidecar: upgraded to support HTTP/3
Network communication: upgraded
```

This reduces the need to rewrite application code.

---

### Example 5: Service Mesh Request Flow

```text id="g1i75i"
Client Service
→ Local Sidecar Proxy
→ Remote Sidecar Proxy
→ Target Service
```

The sidecars may handle:

* TLS
* Routing
* Retry
* Load balancing
* Metrics
* Logs
* Tracing

---

### Example 6: Tracing Across Services

A request travels through several services.

```text id="usfurw"
Service A → Service B → Service C → Service D
```

A sidecar can attach a trace ID:

```text id="7kr612"
trace-id: abc-123
```

Every sidecar forwards the same trace ID, making it easier to see the full request path.

---

### Example 7: Security Centralization

Without sidecar:

```text id="ikdt0q"
Every service must implement TLS correctly.
```

With sidecar:

```text id="0jwexw"
Sidecar handles TLS for every service.
```

This centralizes security behavior.

---

### Example 8: Language-Agnostic Microservices

A system may contain services written in different languages.

```text id="u3cquf"
User Service: Python
Payment Service: Java
Chat Service: Node.js
Recommendation Service: Go
```

With sidecars, each service can still communicate through the same mesh without each language needing identical communication libraries.

---

### Example 9: Sidecar Container in Kubernetes

In Kubernetes, the application container and sidecar container may run in the same pod.

```text id="l7zqbd"
Pod:
  - Application Container
  - Sidecar Proxy Container
```

They can communicate through localhost.

---

### Example 10: Debugging Proxy Tools

Tools such as Fiddler or Charles Proxy use a similar idea.

They sit between the client and server and inspect traffic.

```text id="ts78am"
Application → Local Proxy → Internet
```

This is useful for debugging and observing HTTP traffic.

---

### Example 11: Sidecar Latency Cost

Without sidecar:

```text id="lpfmky"
Service A → Service B
```

With sidecar:

```text id="n4pcgo"
Service A → Sidecar A → Sidecar B → Service B
```

The extra hops add latency and processing overhead.

---

### Example 12: Sidecar Complexity

A service mesh adds many moving parts:

```text id="a8ybbh"
Application
Sidecar proxy
Control plane
Service discovery
Certificate management
Tracing system
Monitoring system
Routing rules
```

This power comes with operational complexity.
