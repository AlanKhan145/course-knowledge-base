# 035 - HTTPS over TFO with TLS 1.3

**Section:** 04 - Many Ways to HTTPS
**Duration:** 2min

---

## 1. Summary

This lecture introduces a more theoretical HTTPS configuration:

> HTTPS over TCP Fast Open with TLS 1.3

TCP Fast Open, or TFO, is an optimization that allows a client to send data during the TCP connection setup phase.

Normally, with TCP, the client must complete the three-way handshake before sending application data.

With TCP Fast Open, if the client already has a valid TFO cookie from the server, it may send data immediately in the initial TCP `SYN` packet.

In this lecture, the idea is to send the TLS 1.3 `ClientHello` early, together with the TCP Fast Open request.

This could theoretically reduce latency by combining part of the TCP setup and TLS setup.

However, this is mostly discussed as a theoretical exercise. It is not commonly seen in practice, and TCP Fast Open has deployment and security concerns.

---

## 2. Key Concepts

### 2.1 The Problem with Normal TCP

In normal TCP, the client must complete the three-way handshake before sending data.

The normal TCP handshake is:

```text
Client → Server: SYN
Server → Client: SYN-ACK
Client → Server: ACK
```

Only after this handshake is complete can the client send useful data.

For HTTPS, this means:

```text
TCP handshake first
TLS handshake second
HTTP request third
```

This adds latency.

---

### 2.2 What Is TCP Fast Open?

TCP Fast Open is an optimization that allows a client to send data earlier during connection setup.

Instead of waiting for the TCP handshake to finish, the client can include early data inside the initial `SYN` packet.

However, this only works if the client already has a valid TFO cookie from the server.

The TFO cookie proves that the client has communicated with the server before.

---

### 2.3 TFO Cookie

A TFO cookie is generated by the server and given to the client during a previous connection.

On a later connection, the client sends the cookie back to the server.

If the server accepts the cookie, it may allow the client to send data before the normal TCP handshake fully completes.

Important clarification:

TCP Fast Open does not truly resume the old TCP connection.

Instead, it allows early data on a new TCP connection when the client has a valid cookie.

---

### 2.4 Combining TFO with TLS 1.3

In this theoretical setup, the early data sent with TCP Fast Open could be the TLS 1.3 `ClientHello`.

That means the client may send:

```text
SYN + TFO cookie + TLS 1.3 ClientHello
```

The server may then respond with:

```text
SYN-ACK + TLS 1.3 ServerHello
```

This combines part of the TCP handshake and TLS handshake.

---

### 2.5 Why This Is Interesting

The goal is to reduce latency.

Instead of doing this:

```text
1. TCP handshake
2. TLS handshake
3. Send encrypted HTTP data
```

The system tries to do this:

```text
1. Start TCP connection and send TLS ClientHello immediately
2. Server responds with TCP and TLS information
3. Finish connection and continue secure communication
```

This can save time by overlapping the TCP and TLS setup phases.

---

## 3. Important Notes

### This Is Mostly Theoretical

The instructor mentions that this is more of a theoretical example.

It may be technically possible in some environments, but it is not commonly used in the wild.

The main purpose is to understand how latency can be reduced by combining protocol steps.

---

### TCP Fast Open Is Not the Same as QUIC

TCP Fast Open tries to improve TCP startup latency.

QUIC was designed from the beginning to combine transport setup and TLS 1.3 encryption setup.

So while TFO can reduce some latency, QUIC is a cleaner and more modern design for this goal.

---

### TCP Fast Open Has Security and Deployment Concerns

TCP Fast Open is not universally used.

Some reasons include:

* Middlebox compatibility problems
* Security concerns
* Replay-related concerns for early data
* Limited support across systems
* More complex server behavior

Because of this, TFO is not as common as TLS 1.3 or QUIC-based optimization.

---

### The Early Data Could Be TLS ClientHello

In this lecture, the early data is not an HTTP request.

The early data is the TLS 1.3 `ClientHello`.

That means the client is trying to start the TLS handshake earlier, not necessarily send the final encrypted HTTP request immediately.

## 4. Examples

### Example 1: Normal HTTPS over TCP with TLS 1.3

```text
Client                                      Server
  |                                           |
  | -------- SYN --------------------------> |
  | <-------- SYN-ACK ---------------------- |
  | -------- ACK --------------------------> |
  |                                           |
  |          TCP connection established       |
  |                                           |
  | -------- TLS ClientHello --------------> |
  | <-------- TLS ServerHello -------------- |
  | -------- TLS Finished -----------------> |
  |                                           |
  | -------- Encrypted HTTP request -------> |
  | <-------- Encrypted HTTP response ------ |
```

Here, TCP setup and TLS setup happen separately.

---

### Example 2: HTTPS over TCP Fast Open with TLS 1.3

```text
Client                                             Server
  |                                                  |
  | -------- SYN + TFO Cookie + ClientHello ------> |
  | <-------- SYN-ACK + ServerHello ---------------- |
  | -------- ACK + TLS Finished -------------------> |
  |                                                  |
  |          Secure connection established           |
  |                                                  |
  | -------- Encrypted HTTP request --------------> |
  | <-------- Encrypted HTTP response ------------- |
```

Here, the TLS 1.3 handshake starts earlier because the `ClientHello` is sent with the TCP `SYN`.

---

### Example 3: Conceptual Difference

Without TCP Fast Open:

```text
Connect first
Then start TLS
Then send HTTP data
```

With TCP Fast Open:

```text
Start connecting
Send TLS ClientHello immediately
Continue TLS setup
Then send HTTP data
```

---

### Example 4: Why a TFO Cookie Is Needed

The server does not want to accept early data from random spoofed clients.

So the client must prove it has talked to the server before.

That proof is the TFO cookie.

Simplified flow:

```text
First connection:
Server gives client a TFO cookie

Later connection:
Client sends SYN + TFO cookie + early data
```

If the cookie is valid, the server may accept the early data.

## Short Takeaway

TCP Fast Open allows a client to send data during the TCP connection setup phase if it already has a valid cookie from the server.

In theory, HTTPS over TFO with TLS 1.3 could send the TLS `ClientHello` together with the TCP `SYN`.

This overlaps TCP setup and TLS setup, reducing latency.

However, this approach is not commonly used in practice and has deployment and security concerns. It is mainly useful as a mental model for understanding how protocol setup costs can be reduced.
