# 036 - HTTPS over TCP with TLS 1.3 and 0-RTT

**Section:** 04 - Many Ways to HTTPS
**Duration:** 2min

---

## 1. Summary

This lecture explains **HTTPS over TCP with TLS 1.3 and 0-RTT**.

TLS 1.3 already reduces handshake latency compared to TLS 1.2. With **0-RTT**, TLS 1.3 can reduce latency even further when the client has connected to the same server before.

The idea is simple:

If the client and server already had a previous TLS connection, the server may have given the client a reusable secret called a **pre-shared key**, or **PSK**.

On a later connection, the client can use that PSK to send encrypted application data immediately during the TLS handshake.

This means the client can send the `ClientHello` and the HTTP request early, without waiting for the full TLS handshake to complete.

---

## 2. Key Concepts

### 2.1 TLS 1.3 0-RTT Requires a Previous Connection

0-RTT only works if the client has already connected to the server before.

During a previous TLS 1.3 connection, the server can send the client a session ticket.

Later, the client can use that session ticket as a pre-shared key.

This allows the client to attempt session resumption.

---

### 2.2 What Is a Pre-Shared Key?

A pre-shared key, or PSK, is a secret derived from a previous TLS session.

The client does not need to repeat the full key exchange from scratch.

Instead, the client sends a hint that tells the server:

```text id="8xquvu"
I have a previous session with you.
I want to resume it.
Here is the identity of the key I want to use.
```

If the server accepts the PSK, both sides can quickly derive encryption keys again.

---

### 2.3 What Does 0-RTT Mean?

0-RTT means:

> The client can send application data without waiting for a TLS round trip.

The client can send early encrypted data together with the TLS `ClientHello`.

For example:

```text id="vufd1t"
ClientHello + PSK extension + encrypted HTTP GET request
```

The server can then accept the PSK, decrypt the early data, and process the request sooner.

---

### 2.4 TCP Handshake Is Still Required

This lecture is about HTTPS over TCP.

That means TCP is still underneath TLS.

So the TCP three-way handshake still has to happen:

```text id="x56z7g"
Client → Server: SYN
Server → Client: SYN-ACK
Client → Server: ACK
```

0-RTT does not remove the TCP handshake.

It reduces the TLS waiting time after TCP is established.

---

### 2.5 The Main Goal Is to Reduce Waiting

The major enemy here is waiting.

Normally, the client sends something and must wait for the server to respond before sending the next thing.

That waiting time is called round-trip latency.

0-RTT tries to reduce that waiting by allowing the client to send useful encrypted data earlier.

---

## 3. Important Notes

### 0-RTT Saves a TLS Round Trip

Without 0-RTT, the client usually waits for the server’s TLS response before sending encrypted HTTP data.

With 0-RTT, the client sends encrypted early data immediately with the TLS handshake.

Simplified:

```text id="uh8b5z"
TLS 1.3 without 0-RTT:
ClientHello → ServerHello → Then send HTTP request

TLS 1.3 with 0-RTT:
ClientHello + encrypted HTTP request → ServerHello + response
```

---

### The Server Can Accept or Reject 0-RTT

The server does not have to accept early data.

If the server accepts the PSK, it can process the early request.

If the server rejects the PSK or rejects early data, the client may need to resend the request after the normal TLS handshake completes.

---

### 0-RTT Has Replay Risk

0-RTT is fast, but it has an important security tradeoff.

Early data can be replayed by an attacker in some scenarios.

Because of that, 0-RTT should only be used for safe and idempotent requests.

Good candidates:

```text id="7vgnc7"
GET /home
GET /image.png
HEAD /resource
```

Bad candidates:

```text id="59jt4x"
POST /checkout
POST /transfer-money
DELETE /account
PUT /update-balance
```

Requests that change state should usually not use 0-RTT.

---

### 0-RTT Is About Latency, Not Bandwidth

0-RTT does not make the network bandwidth larger.

It simply reduces the amount of time spent waiting before the first useful encrypted request can be sent.

The goal is faster connection startup.

## 4. Examples

### Example 1: TLS 1.3 Without 0-RTT

```text id="4i5wes"
Client                                      Server
  |                                           |
  | -------- TCP SYN ----------------------> |
  | <-------- TCP SYN-ACK ------------------ |
  | -------- TCP ACK ----------------------> |
  |                                           |
  |          TCP connection established       |
  |                                           |
  | -------- TLS ClientHello --------------> |
  | <-------- TLS ServerHello -------------- |
  | <-------- Certificate / Finished -------- |
  | -------- TLS Finished -----------------> |
  |                                           |
  | -------- Encrypted HTTP GET -----------> |
  | <-------- Encrypted HTTP 200 OK -------- |
```

The client waits for the TLS handshake before sending the HTTP request.

---

### Example 2: TLS 1.3 With 0-RTT

```text id="gshmmx"
Client                                                  Server
  |                                                       |
  | -------- TCP SYN -----------------------------------> |
  | <-------- TCP SYN-ACK ------------------------------- |
  | -------- TCP ACK -----------------------------------> |
  |                                                       |
  |          TCP connection established                    |
  |                                                       |
  | -------- ClientHello + PSK + Early HTTP GET --------> |
  | <-------- ServerHello + Finished + HTTP Response ----- |
  | -------- Finished ----------------------------------> |
  |                                                       |
```

The client sends the HTTP request earlier, together with the TLS resumption attempt.

---

### Example 3: Previous Connection Creates a Session Ticket

First connection:

```text id="9sx3j7"
Client connects to server normally.
TLS 1.3 handshake completes.
Server sends a session ticket.
Client stores the ticket.
```

Later connection:

```text id="4etxzw"
Client uses the session ticket as a PSK identity.
Client sends ClientHello + early encrypted data.
Server accepts the PSK and decrypts the early data.
```

---

### Example 4: Safe 0-RTT Request

A safe request may look like this:

```http id="33vxqf"
GET /static/logo.png HTTP/1.1
Host: example.com
```

If this request is replayed, it does not change server state.

So it is safer to use with 0-RTT.

---

### Example 5: Unsafe 0-RTT Request

An unsafe request may look like this:

```http id="m5nhqs"
POST /payment HTTP/1.1
Host: bank.com

amount=1000
```

If this request is replayed, money could be transferred multiple times.

This kind of request should not use 0-RTT.

## Short Takeaway

HTTPS over TCP with TLS 1.3 and 0-RTT allows a returning client to send encrypted application data earlier.

The client uses a pre-shared key from a previous connection and sends early data together with the TLS `ClientHello`.

This reduces TLS round-trip waiting time and improves startup latency.

However, TCP is still required, and 0-RTT has replay risks. It should only be used carefully, usually for safe idempotent requests such as `GET`.
